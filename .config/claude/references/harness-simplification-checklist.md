# Harness Simplification Checklist

> Build to Delete 原則を実運用に落とすための定期チェックリスト。
> モデル更新時、または四半期ごとにこのチェックリストを実行する。

---

## トリガー

以下のいずれかが発生したとき、このチェックリストを実行する:

1. **モデル更新**: Claude / Codex / Gemini のメジャーバージョンアップ
2. **四半期レビュー**: 前回の簡素化チェックから3ヶ月経過
3. **複雑さの兆候**: 新しい hook/script/agent の追加が3個以上/月を超えた

## Future-Proofing Test

> 「モデルが改善されたとき、ハーネスの複雑さを追加せずに性能がスケールするか？」

各コンポーネントについて以下を問う:

| 質問 | Yes → | No → |
|------|-------|------|
| このコンポーネントはモデルの弱点を補償しているか？ | 次世代モデルで不要になる可能性がある。削除候補としてマーク | インフラとして必要。維持 |
| モデルを最新に差し替えたとき、このコンポーネントなしで同等の品質が出るか？ | 削除して A/B テスト | 維持 |
| このコンポーネントを削除したとき、何が壊れるか具体的に説明できるか？ | 維持（壊れるものがある） | 削除候補（壊れるものが不明 = 不要の可能性） |

## コンポーネント別チェック

### 1. Hooks (`settings.json` hooks)

- [ ] 各 hook の発火頻度を確認（OTel データまたは git log）
- [ ] 直近30日で一度も発火していない hook → 削除検討
- [ ] モデルが自力で対処できるようになった制約 → 削除検討
- [ ] 例: search-first-gate → モデルが検索を自発的に行うなら不要

### 2. Scripts (`scripts/`)

- [ ] 各スクリプトの呼び出し元を確認（hook / skill / 手動）
- [ ] 呼び出し元が存在しないスクリプト → 削除
- [ ] hook から呼ばれるがモデル改善で不要になったもの → hook と共に削除

### 3. Agents (`agents/*.md`)

- [ ] 直近30日で起動実績のないエージェント → 統合または削除検討
- [ ] 類似スコープのエージェントが3つ以上 → 統合検討
- [ ] description が曖昧で routing ミスが多いもの → description 改善 or 削除

### 4. Skills (`skills/*/SKILL.md`)

- [ ] 直近30日で起動実績のないスキル → 削除検討
- [ ] description トークン総量を確認（目安: 全スキル合計 5K tok 以下）
- [ ] モデルが instruction なしで同等の出力を出せるスキル → 削除候補

### 5. References (`references/*.md`)

- [ ] 各 reference の参照元を確認（CLAUDE.md / skill / agent）
- [ ] 参照元がないもの → 削除
- [ ] CLAUDE.md に昇格すべき重要ルール → 昇格して reference 削除

### 6. MCP Servers (`.mcp.json`)

- [ ] 直近30日で呼び出しのないサーバー → 無効化検討
- [ ] ネイティブツールと重複するもの → MCP 側を削除

## 判定基準

| 判定 | 条件 | アクション |
|------|------|----------|
| **削除** | 30日未使用 + モデル改善で代替可能 | 即座に削除。git history で復元可能 |
| **簡素化** | 使用中だが過剰に複雑 | ロジックを最小限に削減 |
| **統合** | 類似コンポーネントが複数存在 | 1つに統合 |
| **維持** | 必要かつ適切な粒度 | 変更なし |
| **監視** | 判断保留。次回チェックで再評価 | 使用頻度のトラッキングを開始 |

## 実行手順

1. OTel セッションデータまたは git log からコンポーネント使用頻度を取得
2. 上記チェックリストを全コンポーネントに適用
3. 削除/簡素化候補を一覧化
4. A/B テスト（可能な場合）で削除の影響を検証
5. 削除を実行し、変更を `/review` でレビュー
6. 結果を `docs/research/` に記録

## Co-evolution: ツール定義の安定性

> モデルは特定のハーネスとセットで post-training されている。
> ツール定義（名前、description、パラメータ）の変更はモデルの性能退行を引き起こしうる。

### チェック項目

- [ ] ツール名やパラメータ名を変更する場合、モデルが学習済みのパターンを壊さないか確認
- [ ] description の大幅な書き換えは、モデルのツール選択精度に影響する可能性がある
- [ ] ツール削除時は、モデルがそのツールを「呼ぼうとして失敗する」パターンが増えないか監視
- [ ] 新ツール追加時は、既存ツールとのスコープ重複を確認（~10 overlapping tools が分割閾値）

**原則**: ツール定義は API contract と同様に扱う。破壊的変更は避け、段階的に移行する。

## Manus の教訓

> Manus は6ヶ月で5回リビルドし、毎回複雑さを削減した。
> - Complex tool definitions → general shell execution
> - "Management agents" → simple structured handoffs

**原則**: ハーネスに機能を追加する前に「モデル改善でこれは不要にならないか？」を問う。
追加するなら「何が改善されればこれは不要になるか？」を記録する。
