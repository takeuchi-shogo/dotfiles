# Convention Pinning Guide

> Schwartz (Harvard) の "Vibe Physics" プロジェクト (2026-03) で頻発した問題:
> Claude がプロジェクト固有の convention を、教科書や一般的なデフォルトに無意識に戻す。
> 明示的に指定された非標準の命名規約・API 規約・ドメイン用語が drift する。

---

## 問題: Convention Drift

LLM は学習データの多数派パターンに引きずられる。プロジェクト固有の convention が
一般的なパターンと異なる場合、セッションが長くなるほど、または新しいファイルを生成するほど
デフォルトに戻る傾向がある。

### Drift が起きやすい領域

| 領域 | 例 |
|------|-----|
| **命名規約** | snake_case を指定しているのに camelCase で生成 |
| **ディレクトリ構造** | プロジェクト固有の配置ルールを無視して一般的な配置にする |
| **エラーハンドリング** | プロジェクトの error 型を使わず標準の Error を返す |
| **import パス** | エイリアスを使わず相対パスで書く |
| **ドメイン用語** | プロジェクト固有の用語を一般的な表現に言い換える |
| **コミット規約** | プロジェクトの commit format を忘れて素の形式で書く |

## 防止策

### 1. Convention は CLAUDE.md / rules/ に明示する

暗黙の convention は drift する。重要な convention は明文化する。

- 言語固有: `rules/{lang}.md`
- プロジェクト共通: `rules/common/` または プロジェクト CLAUDE.md
- ドメイン用語: `references/` に用語集を配置

### 2. 既存コードからの convention 読み取り

新しいコードを書く前に、同じディレクトリの既存ファイルを読んでパターンを確認する。

**チェックリスト**（新規ファイル作成時）:
- [ ] 同ディレクトリの既存ファイルを 2 つ以上読んだか
- [ ] 命名パターン（変数、関数、ファイル名）を確認したか
- [ ] import スタイルを確認したか
- [ ] エラーハンドリングパターンを確認したか

### 3. Drift 検出の自問

コードを書いた後、以下を自問する:

- 「このパターンは既存コードと一致しているか？」
- 「一般的なデフォルトではなく、このプロジェクトの方式に従っているか？」
- 「CLAUDE.md や rules/ で指定された convention に反していないか？」

### 4. レビューでの Drift チェック

`/review` 時に code-reviewer が確認すべき項目:

- 新規コードが既存コードの convention と一致しているか
- プロジェクト固有の型・ユーティリティが使われているか（標準ライブラリの再発明をしていないか）

## 適用範囲

本ガイドは全プロジェクトに適用される一般原則。
プロジェクト固有の convention 定義は各プロジェクトの CLAUDE.md / rules/ に記載する。
