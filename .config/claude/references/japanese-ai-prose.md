# 日本語成果物の脱・AI 臭ルール

AI が生成する**日本語の文章成果物**（PR 概要・Issue 本文・memo・commit body・ドキュメント）から AI 臭を抜く。

> **concise.md との違い（重要）**: これは brevity（極端な圧縮）ではない。`concise.md` の genshijin モード（体言止め・助詞削除・一文一事実）は会話応答を最短にする指示で、成果物には適さない。成果物は**自然で完全な文**のまま、AI 臭だけを抜く。読みやすさ・完全性を犠牲にしない。

## 抜くパターン

| 抜く | 直す |
|------|------|
| 冗長な動詞句「〜することができる」 | 「〜できる」 |
| 名詞化＋する「〜の向上を実現する」「〜を行う」 | 直接動詞「速くなる」「〜する」 |
| 前置き「重要なのは」「注目すべきは」「〜という点が挙げられる」 | 削って直接述べる |
| ヘッジ「〜と考えられる」「〜と言えるだろう」「〜かもしれない」 | 断定できるなら断定する（不確実なら「未確認」と明示） |
| 機械翻訳調の受動態「〜がなされる」「〜が行われる」 | 行為者を主語に「誰が〜する」 |
| 過剰な接続詞の連打「また」「さらに」「そして」 | 間引く。関係が自明なら接続詞を消す |
| 機械的な三点リスト（常に 3 項目で揃える癖） | 項目数は内容で決める。2 項目で済むなら 2 項目 |
| 空疎な強調「非常に」「極めて」「大幅に」が中身なく付く | 具体的な数値・事実に置換、または削除 |

## Before / After

> Before: 「このアプローチを採用することによって、パフォーマンスの大幅な向上を実現することができます。重要なのは、適切な設計を行うことです。」
>
> After: 「この方法で速くなる。鍵は設計だ。」

> Before: 「本 PR では、認証処理の改善が行われ、エラーハンドリングの強化もなされています。」
>
> After: 「この PR で認証処理を改善し、エラーハンドリングも強化した。」

## 適用しない場面

- **コード / diff / エラーメッセージ / ログ**: 原文のまま
- **技術仕様・API 契約・受入基準**: 曖昧化しない。完全文で厳密に
- **レビュー・検証結果の報告**: 「何を検証し、何が通り、何が落ちたか」は痩せさせない
- **セキュリティ警告・破壊的操作の確認**: 完全文で明示
