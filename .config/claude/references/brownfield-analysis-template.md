# Brownfield 分析テンプレート

既存コードベースへの変更時に使用する体系的分析手順。
AI-DLC の reverse-engineering.md（12ステップ分析）を dotfiles 向けに適応。

`/rpi` の Research フェーズ、または新規プロジェクト参加時に参照する。

---

## 分析チェックリスト

### 必須（M/L タスク）

- [ ] **パッケージ/モジュール構造の把握**: ディレクトリ構成、エントリポイント、主要モジュールの責務
- [ ] **変更対象のビジネスロジック概要**: 対象機能が何をしているか1文で要約
- [ ] **影響する API エンドポイント/インターフェース一覧**: 変更が外部に露出する部分
- [ ] **データモデル/スキーマの確認**: 関連するテーブル、型定義、状態管理

### 推奨（L タスク）

- [ ] **外部依存関係マッピング**: 外部 API、サードパーティライブラリ、マイクロサービス間通信
- [ ] **コーディング規約の検出**: 命名規則、エラーハンドリングパターン、ディレクトリ構成ルール
- [ ] **テスト戦略と現在のカバレッジ**: テストフレームワーク、テスト配置、カバレッジ率
- [ ] **静的モデル**: コンポーネントの責務と関係図（Mermaid で図示）
- [ ] **動的モデル**: 主要ユースケースのシーケンス図（Mermaid で図示）

### オプション（特定状況で有用）

- [ ] **インフラ構成の確認**: デプロイ先、CI/CD パイプライン、環境変数
- [ ] **セキュリティ境界の特定**: 認証/認可の仕組み、シークレット管理方法
- [ ] **パフォーマンス特性**: ボトルネック、キャッシュ戦略、非同期処理パターン

## 出力フォーマット

分析結果は Plan ファイルの Research セクションに記録する:

```markdown
## Research: Brownfield 分析

### 構造
- エントリポイント: `src/index.ts`
- 主要モジュール: auth/, api/, db/
- テストフレームワーク: vitest

### 変更対象
- `src/api/users.ts` — ユーザー CRUD エンドポイント
- `src/db/schema.ts` — users テーブル定義

### 影響範囲
- API: `POST /api/users`, `GET /api/users/:id`
- 依存モジュール: auth/ (認証チェック), db/ (クエリ)

### 規約
- 命名: camelCase (変数), PascalCase (型)
- エラー: Result<T, E> パターン
- テスト: `*.test.ts` で colocate
```

---

## 使い方

1. `/rpi` の Research フェーズ開始時にこのテンプレートを参照
2. タスク規模に応じたチェックリストを実行
3. 結果を Plan ファイルに記録
4. Plan フェーズで分析結果を活用して実装計画を策定
