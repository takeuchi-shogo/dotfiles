# ハーネス政体比較: Claude Code vs Codex

> 出典: wquguru/harness-books — 「殊途同归，还是各表一枝」

## 核心フレームワーク

Claude Code と Codex は同じ前提を共有する:
- モデルは不安定部品
- ハーネスが秩序の源泉
- Prompt だけでは制御できない

**分岐点は「秩序をどの層に配置するか」**。

---

## 二つの政体

### Claude Code: 運行時共和制 (Runtime Republic)

秩序は **Query Loop の現場判断** から生まれる。

| 特徴 | 実装 |
|------|------|
| **制御面** | 動態的拼装（`systemPrompt.ts` で条件注入） |
| **連続性** | Query Loop の状態機 (`query.ts` の跨ターン状態) |
| **本地規則** | `CLAUDE.md` — 会話現場に読み込まれる「現場公告板」 |
| **ツール権限** | 現場調度 (`toolOrchestration.ts` の並行安全判定) |
| **強み** | 柔軟性。現場変化に即応できる |
| **弱み** | 長会話で指令失真・意味稀釈のリスク |

**比喩**: 経験から長出した制御力。現場編導。

### Codex: 控制面立憲制 (Constitutional Control Plane)

秩序は **構造化された fragment と policy** から生まれる。

| 特徴 | 実装 |
|------|------|
| **制御面** | Instruction Fragment (`fragment.rs` で型・境界・包裹規則) |
| **連続性** | Thread / Rollout / State の明示的構造 |
| **本地規則** | `AGENTS.md` — 作用域と優先度の継承関係を明示 |
| **ツール権限** | ExecPolicy (`execpolicy/src/lib.rs` で宣言的ポリシー) |
| **強み** | 可調試性。規則の由来と適用範囲が明確 |
| **弱み** | 構造維持コスト。型・マーカー・シリアライゼーションの維持が必要 |

**比喩**: 制度設計から長出した制御力。制度秘書。

---

## 委譲時の判断軸

「どちらが優れているか」ではなく「何を恐れるか」で選ぶ。

| 主要矛盾 | 推奨先 | 理由 |
|-----------|--------|------|
| 長会話で指令が失真する | Claude Code | runtime heartbeat が現場で指令を再構築 |
| 規則の由来・適用範囲が不明確 | Codex | fragment 化で規則が一等オブジェクト |
| 現場変化が激しい | Claude Code | 動態装配が条件注入に強い |
| 制度化・プログラム化を進めたい | Codex | 構造化 fragment → precedence/merge rule に拡張可能 |

---

## 自分のハーネスを設計するとき

1. **主要矛盾を先に決める**: 現場失控 or 制度失真
2. **片方を先に固める**: 両方同時は「両頭不徹底」のリスク
3. **安定後に他方を補う**: runtime heartbeat が安定 → 構造化を追加、またはその逆
