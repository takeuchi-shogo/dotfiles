# 認証・認可 Archetype

> repair-routing.md からの修復先。このドメインで繰り返しミスが発生した場合に参照・更新する。

## 定義

- **スコープ**: 認証フロー・認可制御・セッション管理・RBAC に関わる変更
- **典型的なファイルパス例**: `auth/`, `middleware/auth*.go`, `*_auth*.go`, `permission*.go`, `rbac*.go`, `handler/login.go`

## 既知の落とし穴

| 落とし穴 | 症状 | 関連 FM | 防止策 |
|---------|------|---------|--------|
| 認可バイパス | 認証済みでも権限外リソースにアクセス可能 | FM-010 | 認証チェックと認可チェックを独立して実装する |
| トークン漏洩 | JWTをログ/レスポンスに含める | FM-010 | トークンを構造体ではなく不透明な文字列として扱う |
| RBAC 設定ミス | ロール追加時に既存ポリシーと矛盾 | FM-006 | ロール変更時は全エンドポイントの権限マトリクスを再検証 |
| セッション固定 | ログイン後にセッションIDを再生成しない | FM-010 | 認証成功時は必ずセッションIDを再発行する |
| middleware chain の順序誤り | 認可が認証より先に実行される | FM-006 | 認証 → 認可の順序をミドルウェア登録コードで強制する |

## 不変条件

- 認証（誰か）→ 認可（何ができるか）の順序は必ず守る
- 最小権限原則: デフォルトは拒否、明示的な許可のみ通す
- トークンには有効期限（`exp` クレーム）を必ず設定する
- パスワードは bcrypt/argon2 等のコスト付きハッシュで保存する（平文・MD5 禁止）
- 認証エラーのメッセージは「ユーザーが存在しない」と「パスワード誤り」を区別しない
- Go の `context` にユーザー情報を載せるときは型付きキーを使う（`string` キー禁止）

## テスト戦略

| テスト種別 | 目的 | 優先度 |
|-----------|------|--------|
| 権限境界テスト | ロールAがロールBのリソースにアクセスできないことを確認 | Critical |
| ネガティブアクセステスト | 未認証リクエストが 401/403 を返すことを確認 | Critical |
| セッション管理テスト | ログイン後のセッションID変更、ログアウト後の無効化を確認 | High |
| トークン期限切れテスト | 期限切れトークンが拒否されることを確認 | High |
| RBAC マトリクステスト | 全ロール×全エンドポイントの権限組み合わせを網羅 | Medium |
