# 入力検証 Archetype

> repair-routing.md からの修復先。このドメインで繰り返しミスが発生した場合に参照・更新する。

## 定義

- **スコープ**: API リクエスト・CLI 引数・設定ファイル・ファイルアップロード等の外部入力を受け付ける処理
- **典型的なファイルパス例**: `handler/`, `validator/`, `middleware/validate*.go`, `usecase/*/input.go`, `cmd/`

## 既知の落とし穴

| 落とし穴 | 症状 | 関連 FM | 防止策 |
|---------|------|---------|--------|
| 境界値チェック漏れ | 0・最大値・空文字・負数でパニック or 不正動作 | FM-005 | テーブル駆動テストで境界値（min, max, min-1, max+1）を網羅 |
| クライアントサイドのみの検証 | JS バリデーションをバイパスされてサーバーが受け入れる | FM-005 | サーバーサイドで必ず独立して検証する |
| sanitization 忘れ | HTML/SQL 特殊文字がそのまま処理される | FM-005, FM-010 | 入力正規化（trim/normalize）→ 検証の順序を守る |
| NULL/ゼロ値の取り違え | 未入力と明示的なゼロを区別できない | FM-001 | ポインタ型またはラッパー型で「未設定」を表現する |
| denylist ベースのフィルタ | ブラックリスト漏れで不正値が通過 | FM-005 | allowlist（許可リスト）を基本とする |

## 不変条件

- 信頼境界（サービス入口）では必ずサーバーサイドで検証する
- allowlist > denylist: 許可する値を明示し、それ以外を拒否する
- 入力の順序: 正規化（trim/decode）→ バリデーション → 使用
- バリデーションエラーは具体的なフィールド名とメッセージを返す
- ファイルアップロードは MIME タイプ・サイズ・拡張子を独立して検証する

## テスト戦略

| テスト種別 | 目的 | 優先度 |
|-----------|------|--------|
| 境界値テスト | min/max/空/NULL/境界±1 を網羅したテーブル駆動テスト | Critical |
| 型変換エラーテスト | 不正な型（文字列→整数等）で適切なエラーを返すことを確認 | High |
| sanitization テスト | HTML/SQL 特殊文字がエスケープされることを確認 | High |
| fuzz テスト | ランダム入力でパニックしないことを確認（`go-fuzz` / `testing.F`） | Medium |

## Change Surface 連携

- **対応する Change Surface**: 「バリデーション」(Medium)
- **推奨 preflight**: edge-case-hunter（境界値・NULL パス・型変換エラーの網羅）
- `change-surface-preflight.md` には「バリデーション」専用パターンはないが、`handler/` や `api/` パターンで補足される
