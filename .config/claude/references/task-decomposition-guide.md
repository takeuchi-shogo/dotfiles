# Task Decomposition Guide

タスクの粒度を最適化するためのガイドライン。
INVEST 原則をベースに、Claude Code エージェントハーネス文脈での適用を定義する。

## 原則: 1タスク = 1つの検証可能な変更

タスクが大きすぎると:
- AI の生成精度が低下する（コンテキスト過負荷）
- レビューの品質が低下する（変更範囲が広すぎて焦点がぼやける）
- 失敗時のロールバックが困難になる

## INVEST チェックリスト

| 基準 | 意味 | 適用 |
|------|------|------|
| **I**ndependent | 他タスクと独立して完了できる | 依存がある場合は順序を明示 |
| **N**egotiable | 詳細は実装時に調整可能 | 過度に詳細な仕様を前もって決めない |
| **V**aluable | 単体で価値を提供する | 「準備だけ」のタスクを避ける |
| **E**stimable | 見積もり可能なサイズ | S/M/L で見積もれないなら分割 |
| **S**mall | 1セッションで完了できる | 目安: 変更ファイル10以下 |
| **T**estable | 完了条件が明確 | acceptance criteria を定義 |

## 分割の目安

| シグナル | アクション |
|----------|-----------|
| 変更ファイル 10 超の見込み | PR Split Plan を作成（spec SKILL.md 準拠） |
| 3つ以上の独立した変更を含む | 個別タスクに分割 |
| 「A して、ついでに B も」 | A と B を分離 |
| テストと実装が別の関心事 | テスト先行タスク + 実装タスクに分割 |

## ハーネス文脈での適用

- **S 規模**: 分割不要。即実装
- **M 規模**: Plan で粒度確認。必要なら 2-3 タスクに分割
- **L 規模**: PR Split Plan 必須。各サブタスクが INVEST を満たすか検証
- **改善サイクル** (`/improve`): 1回の改善で変更は最大3ファイル（improve-policy.md 準拠）

## アンチパターン

| NG | 理由 |
|----|------|
| 「リファクタリングと新機能を同時に」 | 変更の意図が混在し、レビュー・ロールバックが困難 |
| 「テストは後で書く」 | テスト不在の変更がマージされ、品質が低下 |
| 「小さすぎる分割」（1行ずつ） | オーバーヘッドが価値を上回る |
| 「依存関係を無視した並列化」 | 結合時にコンフリクト・不整合が発生 |
