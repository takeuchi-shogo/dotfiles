# Task Decomposition Guide

タスクの粒度を最適化するためのガイドライン。
INVEST 原則をベースに、Claude Code エージェントハーネス文脈での適用を定義する。

## 原則: 1タスク = 1つの検証可能な変更

タスクが大きすぎると:
- AI の生成精度が低下する（コンテキスト過負荷）
- レビューの品質が低下する（変更範囲が広すぎて焦点がぼやける）
- 失敗時のロールバックが困難になる

## INVEST チェックリスト

| 基準 | 意味 | 適用 |
|------|------|------|
| **I**ndependent | 他タスクと独立して完了できる | 依存がある場合は順序を明示 |
| **N**egotiable | 詳細は実装時に調整可能 | 過度に詳細な仕様を前もって決めない |
| **V**aluable | 単体で価値を提供する | 「準備だけ」のタスクを避ける |
| **E**stimable | 見積もり可能なサイズ | S/M/L で見積もれないなら分割 |
| **S**mall | 1セッションで完了できる | 目安: 変更ファイル10以下 |
| **T**estable | 完了条件が明確 | acceptance criteria を定義 |

## 分割の目安

| シグナル | アクション |
|----------|-----------|
| 変更ファイル 10 超の見込み | PR Split Plan を作成（spec SKILL.md 準拠） |
| 3つ以上の独立した変更を含む | 個別タスクに分割 |
| 「A して、ついでに B も」 | A と B を分離 |
| テストと実装が別の関心事 | テスト先行タスク + 実装タスクに分割 |

## ハーネス文脈での適用

- **S 規模**: 分割不要。即実装
- **M 規模**: Plan で粒度確認。必要なら 2-3 タスクに分割
- **L 規模**: PR Split Plan 必須。各サブタスクが INVEST を満たすか検証
- **改善サイクル** (`/improve`): 1回の改善で変更は最大3ファイル（improve-policy.md 準拠）

## アンチパターン

| NG | 理由 |
|----|------|
| 「リファクタリングと新機能を同時に」 | 変更の意図が混在し、レビュー・ロールバックが困難 |
| 「テストは後で書く」 | テスト不在の変更がマージされ、品質が低下 |
| 「小さすぎる分割」（1行ずつ） | オーバーヘッドが価値を上回る |
| 「依存関係を無視した並列化」 | 結合時にコンフリクト・不整合が発生 |

## M/L 規模タスクの事前見積もりプロンプト

> 出典: erukiti「実践フルAIコーディングのための考え方とノウハウ」(Zenn 2024-12)

M/L 規模のタスクを Plan ステージで委譲する前に、以下を **列挙させて** 規模感と矛盾を事前検出する。LLM の「勝手な推測・予測・憶測」を抑制する効果がある。

```
以下のタスクに着手する前に、変更内容を見積もってください。
**勝手な推測・予測・憶測は絶対禁止**。コード・データを直接確認した結果だけを列挙してください。

タスク: {タスク説明}
関連設計文書: {paths}

以下を列挙してください:
1. **編集対象ファイル**: ファイルパスと該当行番号 (新規追加なら追加予定位置)
2. **関連 DB テーブル / カラム**: 読み書き対象 (該当なしなら "該当なし")
3. **ログ出力**: 追加・変更されるログメッセージ
4. **外部 I/O**: HTTP/RPC/file/queue など (該当なしなら "該当なし")
5. **疑問点・矛盾点**: 仕様間で不整合がある箇所
6. **リファクタが必要な箇所**: 既存コードで先に整理が必要な部分
7. **フェイズ分割の可否**: 1 タスクで完結するか、フェイズ分割すべきか (理由付き)

回答は箇条書きで、推測ではなく事実だけを書いてください。
```

**期待効果**:
- PR サイズが事前に可視化される (10 ファイル超なら分割誘導)
- 仕様矛盾が実装前に検出される
- リファクタ先行の必要性が明示される

## テスト変更と実装変更のフェイズ分離プロンプト

> 出典: 同上。リファクタ観点でテストの変更と実装の変更を必ず分離させる。

リファクタや挙動変更を伴うタスクでは、**テスト変更フェイズと実装変更フェイズを分離**するよう明示する:

```
本タスクをフェイズ分割してください。各フェイズは独立してマージできる単位とすること。

**ルール**:
- ソースコードを変更するフェイズでは、ユニットテストの変更を禁止する
- ユニットテストを変更するフェイズでは、ソースコードの変更を禁止する
- 各フェイズ完了時に既存テストが全て通ること (テスト先行フェイズでは既存実装で新テストが赤になることを許容)
- フェイズ間の順序と理由を明示する

回答形式:
- Phase 1: {テスト変更 or 実装変更} — 理由: ...
- Phase 2: ...
- (以下同様)
```

**期待効果**:
- 「テストに合わせて実装をいじる」「実装に合わせてテストを書き換える」自家中毒の防止
- 各フェイズが独立してレビュー・ロールバック可能
- リファクタ先行と新機能実装の混在を防ぐ
