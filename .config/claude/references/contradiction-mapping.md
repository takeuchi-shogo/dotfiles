# Contradiction Mapping — 矛盾検出と解決

> 知識ベース内の矛盾する知見を検出・解決するフレームワーク。
> Spark 記事の矛盾マッピング概念を AutoEvolve の知識管理に統合する。

## 矛盾検出条件

以下の **両方** を満たす場合、矛盾として検出する:

1. **同一トピック**: 同じファイル・パターン・カテゴリに関する知見
2. **方向性逆転**: 一方が「推奨」し他方が「非推奨」、または数値指標が逆方向

## 矛盾の種類

| 種類 | 説明 | 例 |
|------|------|-----|
| 直接矛盾 | 同一事項に対する正反対の指示 | 「常にモックを使え」vs「実DBでテストせよ」|
| 条件矛盾 | 異なる条件下で有効な知見が無条件に記述 | 「小規模なら monolith」vs「microservices 推奨」|
| 時系列矛盾 | 古い知見が新しい状況に合わない | 旧バージョンの API 仕様に基づくルール |

## 解決フロー（5ステップ）

1. **検出**: Garden フェーズで同一トピックの知見を比較
2. **分類**: 直接矛盾 / 条件矛盾 / 時系列矛盾 のいずれかに分類
3. **境界条件の特定**: どの条件下でどちらが有効かを特定
4. **統合**: `boundary_condition` フィールドを付与して両方を保持、または片方を降格
5. **検証**: 統合後の知見が既存ルール・テストと整合することを確認

## boundary_condition フィールド

矛盾が条件付きで両立する場合、知見エントリに `boundary_condition` を追加する:

```json
{
  "rule": "テストでモックを使用する",
  "boundary_condition": "ユニットテストのみ。統合テストでは実DBを使用",
  "valid_when": "unit_test",
  "superseded_by": null
}
```

- `boundary_condition`: 自然言語での適用条件
- `valid_when`: 機械可読な条件タグ
- `superseded_by`: 時系列矛盾の場合、新しい知見の ID

## Garden フェーズとの統合

- 矛盾検出は Garden フェーズの Quality Gate で実行
- 検出された矛盾は `[CONTRADICTION]` タグ付きで `/improve` ダッシュボードに表示
- 自動解決は行わない。ユーザーに判断を委ねる
