# Agent Conflict Patterns

並列エージェント実行時に発生しうる矛盾パターンと、Opus（コーディネーター層）が結果統合時に検証すべきチェックリスト。

## 矛盾パターンカタログ

| # | パターン | 例 | 検出方法 |
|---|---------|---|---------|
| 1 | **同一ファイル矛盾編集** | Agent A が関数を追加、Agent B が同じファイルの別関数を削除 | worktree 使用で物理的に回避。同一ブランチの場合は diff の重複箇所を確認 |
| 2 | **相反する推奨** | レビューア A が「この関数を分割すべき」、B が「この関数は十分シンプル」 | コーディネーターが推奨を突合せし、根拠の強さで判断 |
| 3 | **前提の不整合** | Agent A が古いコード状態を前提に分析、Agent B が最新を前提 | 並列起動時にベースコミットを揃える（worktree or 同一 HEAD） |
| 4 | **スコープ侵食** | 実装エージェントがレビュー範囲のコードも変更 | agent 契約（Analyze-only / Implement）でツール制限 |
| 5 | **リソース競合** | 2つのエージェントが同じ外部 API を同時に叩いてレート制限に到達 | MCP ツール使用の場合、順次実行を検討 |

## コーディネーター検証チェックリスト

並列エージェントの結果を統合する際、以下を確認する:

- [ ] **ファイル重複**: 複数エージェントが同一ファイルを変更していないか
- [ ] **推奨の整合性**: 矛盾する推奨がないか。ある場合は根拠を比較し、より具体的な evidence を持つ方を採用
- [ ] **前提の一貫性**: 各エージェントが同じコードベース状態を前提にしているか
- [ ] **スコープ遵守**: 各エージェントが契約範囲内で作業しているか
- [ ] **副作用の交差**: Agent A の変更が Agent B の前提を壊さないか

## ガードレール文言

`/dispatch` や Agent 並列実行時のプロンプトに以下を注入する:

```
## Parallel Execution Guard

あなたは他のエージェントと並列で作業しています。以下を遵守してください:

1. 指定されたスコープ外のファイルを変更しない
2. 他エージェントの作業領域と重複する可能性がある場合、変更理由を明示する
3. 推奨を出す際は、根拠（ファイル名:行番号、エラーメッセージ等）を必ず添える
4. 外部 API やリソースへのアクセスは最小限に留める
```

## 適用タイミング

| シナリオ | 対応 |
|---------|------|
| `/dispatch` で複数 Worker 起動 | ガードレール文言をプロンプトに注入 |
| Agent 並列実行（同一セッション） | worktree 分離を推奨 |
| `/review` で複数レビューア並列 | コーディネーターが結果統合時にチェックリスト実行 |
| サブエージェント BG 実行 | 完了通知受信後にチェックリスト実行 |

## Build to Delete

この文書は「Opus の判断力でカバー」するためのリファレンスであり、自動 hook 化はしない。
モデルが並列実行の矛盾を自律的に検出・解決できるようになれば、この文書は不要になる。
