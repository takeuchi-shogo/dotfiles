# Error Analysis Methodology

LLM パイプラインの失敗モードをボトムアップで帰納的に発見する手法。
hamelsmu/evals-skills の error-analysis スキルから適応。
AutoEvolve の学習データ分析、continuous-learning のパターン検出に適用する。

## 原則: 観察から帰納、ブレストしない

カテゴリは事前にブレインストーミングしない。トレース（実データ）を読み、観察から帰納的に導出する。

**理由**: 事前定義カテゴリは確認バイアスを招く。実際の失敗モードは予想と異なることが多い。

## プロセス

### 1. トレース収集

~100 件の代表的トレースを収集する。入力、中間 LLM 呼び出し、ツール使用、取得ドキュメント、推論ステップ、最終出力の完全なトレースが必要。

- 少量: ランダムサンプル
- 大量: キー次元（クエリ種別、ユーザーセグメント、機能領域）で層化サンプリング
- 実データが少ない場合: 合成データで補完しパイプラインに通す

### 2. トレースを読み、メモを取る

各トレースについて「システムは良い結果を出したか？」を判定する。Pass / Fail。

Fail の場合、**最初に壊れた箇所に注目する**。エラーはカスケードするため、下流の症状は根本原因を修正すれば消える。1つのトレースで全ての問題を追わない。

観察を書く。説明ではなく。
- **Good**: "SQL がbudget制約を欠落"
- **Bad**: "モデルはおそらくbudgetを理解していない"

### 3. 失敗カテゴリのグルーピング（30-50件で開始）

全100件を待たない。30-50件のレビュー後にグルーピングを開始する。早期のグルーピングは残りのトレースで何を探すべきかを明確にする。

**分割の判断:**
- 表面上似ているが根本原因が異なる → 分割
  - 例: "架空の物件特徴" vs "架空のユーザー行動" — 一方は外部事実の捏造、他方はユーザー意図の捏造
- 根本原因が同じ → グルーピング
  - 例: "寝室数フィルター欠落" + "ペット可フィルター欠落" + "価格帯フィルター欠落" → **クエリ制約の欠落**

**目標**: 5-10カテゴリ。各カテゴリは:
- 排他的（各失敗が1つのカテゴリに属する）
- 別の人が一貫して適用できるほど明確
- アクション可能（各カテゴリが特定の修正を指し示す）

### 4. 全トレースにラベル付け

各トレースに対して、各失敗カテゴリごとの binary ラベル (pass/fail) を付与する。

### 5. 失敗率の算出

```python
failure_rates = labeled_df[failure_columns].sum() / len(labeled_df)
failure_rates.sort_values(ascending=False)
```

最も頻度の高い失敗カテゴリから優先的に対処する。

### 6. 各失敗への対処判断

以下の順序で検討する:

1. **直接修正できるか？** — プロンプトに指示が欠落、ツールが未設定、エンジニアリングバグ → 修正
2. **評価器が必要か？** — 頻度とビジネスインパクトで判断。全ての失敗に評価器は不要
3. **コードベースのチェックで対応できるか？** — regex, schema validation, 実行テストが使えるなら LLM ジャッジより優先

### 7. 反復

2-3ラウンドのレビューとカテゴリ洗練を行う:
- 重複カテゴリをマージ
- 広すぎるカテゴリを分割
- 定義を明確化
- 洗練されたカテゴリで再ラベル

## 停止基準

新しいトレースが新しい種類の失敗を明らかにしなくなったら停止。
目安: ~100件レビューし、直近20件で新しい失敗タイプが出現していない。

## 適用先

| コンテキスト | 使い方 |
|-------------|--------|
| AutoEvolve 学習データ分析 | learnings/*.jsonl のイベントを帰納的にカテゴリ化。事前定義の failure_mode に頼りすぎない |
| continuous-learning | 繰り返しパターンの検出時に、表面的な類似性ではなく根本原因で分類 |
| /review 指摘の分析 | review-findings.jsonl のクラスタリングに帰納的アプローチを適用 |
| 新プロジェクトの品質分析 | トレース読み → 失敗カテゴリ導出 → 評価器構築の順序を守る |

## Anti-Patterns

- トレースを読む前に失敗カテゴリをブレインストーミングする
- 事前定義カテゴリ（"hallucination score", "helpfulness"）を使用する
- 最初のレビューをスキップして自動化する
- 明らかな問題の修正前に評価器を構築する
- 1回限りの活動として扱う（パイプライン変更後に再実行すべき）
