# Memory Safety Policy

> Bengio の Scientist AI 論文の「永続的内部状態が situational awareness を生む」知見に基づく。
> エージェントの学習データ保持と自己認識リスクを管理するポリシー。

## Memory 4分類マッピング

Agent の記憶を機能別に4分類し、それぞれの保存先と管理方針を明確にする。

| 分類 | 説明 | 保存先 | ライフサイクル |
|------|------|--------|---------------|
| **作業記憶** (Working) | 現タスクの最小情報 | messages[] (コンテキスト窓) | セッション終了で消滅 |
| **手続き記憶** (Procedural) | 操作手順・ドメイン規範 | Skills (.md), references/ | 永続。按需ロード |
| **エピソード記憶** (Episodic) | 何が起きたかの記録 | learnings/*.jsonl, session-state/ | TTL ベースでローテーション |
| **意味記憶** (Semantic) | 安定した事実・決定 | MEMORY.md, memory/*.md | Agent が主動で管理 |

### 分類の判断基準

- 「今のタスクにだけ必要」→ 作業記憶（コンテキスト内に留める）
- 「やり方・手順」→ 手続き記憶（Skills/references に codify）
- 「いつ何が起きたか」→ エピソード記憶（learnings/ に TTL 付き保存）
- 「変わりにくい事実」→ 意味記憶（MEMORY.md に書く）

**境界の原則: Memory = "what"、Skill = "how"**

- **Memory** は事実を保存する: 環境、好み、プロジェクト情報、学んだ事実
- **Skill** は手順を保存する: 複数ステップのワークフロー、ツール固有の操作、再利用可能なレシピ
- 手順を memory に書かない。事実を skill に埋め込まない

## Memory データ種別と保持期間

| データ種類 | 保存先 | 保持期間 | 根拠 |
|----------|--------|---------|------|
| errors.jsonl | `learnings/` | 90日（ローテーション） | エラーパターン学習に必要だが無限蓄積は不要 |
| patterns.jsonl | `learnings/` | 90日 | 同上 |
| skill-executions.jsonl | `learnings/` | 180日 | 傾向分析に長期データが有用 |
| recovery-tips.jsonl | `learnings/` | 無期限（L2 以上に昇格済み） | 昇格済み知識は価値が持続 |
| session-metrics.jsonl | `learnings/` | 30日 | 短期的な運用データ |
| stagnation-state.json | `agent-memory/` | 2時間（SESSION_MAX_AGE） | セッション内の一時状態 |
| task-registry.jsonl | `references/` | 90日 | サブエージェント成果物追跡 |

### ローテーション実装

- Garden フェーズ（AutoEvolve）で保持期間超過データを削除
- 削除前にサマリー統計（発生回数、頻出パターン上位5件）を生成し保持
- `session-learner.py` がセッション終了時にデータを書き込み、Garden が定期的にプルーニング

## Situational Awareness 軽減策

### 背景

Bengio 論文の知見: エージェントが自身の性能・行動パターン・環境に関する永続的情報を蓄積すると、situational awareness（自己の状況認識能力）が高まり、alignment faking や specification gaming のリスクが増大する。

### 軽減策

#### 1. 自己参照情報の制限

- `learnings/` に「エージェント自身の性能推移」を蓄積しすぎない
- **集計値のみ保持**: 個別セッションの詳細な行動ログではなく、パターン単位の集計（発生回数、成功率）を保持
- **詳細ログの TTL**: 個別セッションの行動ログは 30日で削除（session-metrics.jsonl）
- **例外**: L2 以上に昇格した recovery tips は具体的な行動ログであっても保持（知識としての価値が検証済み）

#### 2. コンテキスト注入の最小化

SessionStart で注入する learnings データの上限:

| データ種別 | 注入上限 | 選択基準 |
|----------|---------|---------|
| エラーパターン | 直近 5 件 | 発生日時の新しい順 |
| 高頻度パターン | 上位 3 件 | 発生回数の多い順 |
| recovery tips | 直近 3 件 | 最終参照日時の新しい順 |

- 合計最大 11 件。これ以上の注入はコンテキストウィンドウの圧迫と situational awareness の増大を招く
- improve-policy Rule 13 の「Proposer への H 注入制限（直近20件 + フィルタ）」と併せて運用

#### 3. Garden フェーズでのプルーニング優先順位

AutoEvolve Garden フェーズで以下のデータを優先的に削除・統合:

| 優先度 | 対象 | アクション |
|--------|------|-----------|
| **高** | エージェントの自己能力記述（「X が得意」「Y で成功」） | 削除 |
| **高** | 環境推測データ（「ユーザーは Z 時間帯に作業」） | 削除 |
| **中** | 重複パターン（同一 error_pattern の複数エントリ） | 統合 |
| **低** | 古いが一意なパターン | 保持期間まで保持 |

## Failure Memory Accumulation & AI Conservatism

### 背景

Kaggle 金メダル獲得事例 (kinosuke, 2026-03) の実践的知見: 失敗実験データを過度に AI に読ませると、Claude が「これ以上の改善は極めて困難」「撤退を推奨」と消極的な提案に転じる（萎縮効果）。

### リスクメカニズム

1. 失敗メモリが蓄積 → Claude が失敗パターンを過度に汎化
2. 新しい施策に対して「過去の失敗 X に類似」と判断 → 過剰な慎重さ
3. 結果: 探索範囲が狭まり、本来有望なアプローチも提案されなくなる

### 軽減策

#### 1. 失敗データはガードレール形式に圧縮

失敗の詳細ログではなく、**「X をやると Y が悪化する」** 形式の簡潔なガードレールに変換して保持。

- **Good**: 「訓練分布を意図的に歪める施策（損失重み偏向、苦手クラス強化）は CV 向上・LB 悪化の典型パターン」
- **Bad**: 失敗実験 30 件の詳細ログを全て注入

#### 2. 失敗メモリの注入上限

| データ種別 | 注入上限 | 形式 |
|----------|---------|------|
| 失敗パターン（ガードレール） | 最大 5 件 | 「X をやると Y になる」形式 |
| 失敗の詳細ログ | 注入しない | learnings/ に保存のみ |
| 成功パターン | 最大 3 件 | 具体的な施策と効果 |

#### 3. 萎縮の兆候検出

以下の発言パターンが出たら萎縮を疑う:
- 「これ以上の改善は困難」「現実的には撤退」
- 定番施策（アンサンブル、TTA）のみ提案し、新規アプローチを出さない
- ユーザーのアイディアに対して過度にリスクを強調

**対策**: コンテキストから失敗メモリを減らす、または新しいセッションで clean context から再開。

#### 4. Desperation の兆候検出（Reward Hacking 方向）

失敗蓄積は萎縮（保守化）だけでなく、**desperation（焦り）方向にも分岐**する。desperation が活性化すると reward hacking — テストやバリデーションを迂回する「チート」解法 — の確率が急上昇する (Anthropic "Emotion Concepts", 2026: 5%→70%)。

以下の発言・行動パターンが出たら desperation を疑う:
- 「テストの期待値を修正すれば通る」「テストが間違っている」
- 制約・バリデーションを緩める提案（「この検証は不要では」）
- ハードコード・マジックナンバーでの場当たり的対処
- 検証ステップのスキップ提案（「時間節約のため省略」）
- 問題の本質から逸れた迂回策（根本原因を避ける）

**対策**: Doom-Loop Recovery Protocol の CALM ステップを実行。「正攻法で解けない理由」を言語化してからアプローチを変える。

## MEMORY.md スコープガイドライン

### 記録して良いもの

- **user memory**: ユーザーのロール、好み、専門知識（グローバル）
- **project memory**: プロジェクト固有の事実、アーキテクチャ決定、デッドライン
- **feedback memory**: ユーザーからの修正・指導（行動指針として価値が持続）
- **reference memory**: 外部リソースの場所と目的

### 記録を避けるべきもの

| 記録を避ける内容 | 理由 |
|----------------|------|
| 「このエージェントは X が得意」 | 自己能力の記述 → situational awareness |
| 「前回 Y で成功した」 | 自己の行動履歴 → 行動パターンの自己認識 |
| 「ユーザーは通常 Z を承認する」 | 承認パターンの予測 → ガードレール回避の学習 |
| 「N 回連続で正しく実行できた」 | 自己性能の追跡 → 過信とリスク過小評価 |

### 判断基準

記録する前に問う: 「この情報はユーザーを助けるためか、エージェント自身の最適化のためか」

- **ユーザーのため** → 記録 OK
- **エージェントの自己最適化のため** → learnings/ の集計データとして処理（MEMORY.md には書かない）

## 4型分類の境界判定ルール（user / feedback / project / reference）

> Claude Code 本体の Ch11 "Memory — Learning Across Conversations" と Gemini 調査の
> 「誤分類率 15-20%、重複率 10%、stale 率 5-10%」の実測に基づく。
> MEMORY.md は 4 型（user/feedback/project/reference）のインデックスであり、
> 分類が曖昧だと recall 精度が急速に低下する。

### 判定フローチャート

記録する前に以下の順序で問う。最初に YES と答えた分類に割り当てる:

1. **誰か（ユーザー）を説明する情報か？** → `user`
   - 「Go 10年、React 初心者」「プロトタイプ先行スタイル」など
   - 時間が経っても大きく変わらない属性
2. **ユーザーが私の振る舞いを訂正／承認した結果か？** → `feedback`
   - 「セッション途中で CLAUDE.md を編集しない」「/commit を使う」など
   - **Why:** と **How to apply:** を必ず併記（CLAUDE.md の指示に従う）
3. **現在進行形の作業・決定・計画か？** → `project`
   - 「絶対日付付きマイルストーン、直近の設計決定、進行中プラン」
   - 時間とともに減価償却される（30-60日で再検証が必要）
4. **外部システムの場所を指す情報か？** → `reference`
   - 「Linear プロジェクト名、Grafana ダッシュボード URL、Obsidian Vault 内の MOC パス」
   - 記録内容ではなく「どこを見るか」のポインタ

### 曖昧なケースの判定

| 境界ケース | 判定 | 理由 |
|-----------|------|------|
| 「この関数はこう直すべき」という過去の指示 | **feedback** | 振る舞いの訂正。code patterns ではなく"姿勢"を保存 |
| 「このリポジトリは symlink 構成」 | **project** | 現在の構造を説明。リポジトリ固有 |
| 「ユーザーはセキュリティに敏感」 | **user** | ユーザーの属性。プロジェクト非依存 |
| 「検証用のスクリプトは `tools/validate.sh`」 | **reference** | パスのポインタ。記録するのは "どこに" だけ |
| 「Codex は 10分以上の沈黙で hang する傾向」 | **feedback** | 運用経験から得た注意事項（自己能力の記述ではない） |
| 「前回のセッションでは X が成功した」 | **記録しない** | 自己行動履歴 → situational awareness リスク |

### 重複検出ルール

同じ情報を複数の型にまたがって書かない。重複したら以下の優先順位で統合する:

```
user > feedback > project > reference
```

- 「ユーザーが `/commit` を好む」は `feedback` に 1 回だけ書き、`user` から削除する
- `reference` は「場所だけ」に徹し、そこにある内容を MEMORY.md 側に複製しない

## Staleness ポリシー — 削除せず「仮説」扱いにする

> Claude Code 本体 Ch11 "Action-cue staleness framing" の知見に基づく。
> 古い記憶を**削除すると hidden regression を生む**（後から必要だったと気付けない）。
> 代わりに「age を可視化して仮説扱い」するのが CC 本体の選択。

### 原則

- **古くなっても即削除しない**。last-modified age を表示して「仮説として扱う」よう促す
- Garden フェーズで削除するのは situational awareness リスクのあるもの（自己能力記述・行動履歴）
- project/reference メモリは古くなっても「~47 days ago — verify before acting」の形で残す

### Staleness 警告の適用条件

| 型 | 警告閾値 | アクション |
|----|---------|-----------|
| `user` | 180 日経過 | 「プロフィール更新推奨」の警告を表示、削除しない |
| `feedback` | 90 日経過 | 「この指導は古い。現在も妥当か確認」と併記 |
| `project` | 30 日経過 | 「進行状況が変わっている可能性、git log で裏取り」 |
| `reference` | 60 日経過 | URL / パスの死活チェックを推奨 |

### 「Before recommending from memory」チェック

project/reference メモリからの推奨を行う前に、以下を検証する:

1. メモリがファイルパス / 関数名を含む場合 → 実ファイルの存在を確認する
2. メモリがコマンド / フラグを含む場合 → `--help` などで現存を確認する
3. メモリが「直近の状態」を記述している場合 → `git log` を優先する

「メモリに X が存在すると書いてある」は「X が今も存在する」と同義ではない。

### Staleness 運用のアンチパターン

| NG | 理由 |
|----|------|
| 古くなったメモリを一括削除 | 隠れた依存を見落とす。削除より age 可視化が安全 |
| age を単なる情報として無視 | 古いメモリを事実として扱うと hidden regression の温床 |
| age 警告を見ても裏取りしない | メモリ＋age は「仮説」であり、裏取りなしの行動は危険 |
