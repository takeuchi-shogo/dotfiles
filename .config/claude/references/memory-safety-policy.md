# Memory Safety Policy

> Bengio の Scientist AI 論文の「永続的内部状態が situational awareness を生む」知見に基づく。
> エージェントの学習データ保持と自己認識リスクを管理するポリシー。

## Memory 分類と保持期間

| データ種類 | 保存先 | 保持期間 | 根拠 |
|----------|--------|---------|------|
| errors.jsonl | `learnings/` | 90日（ローテーション） | エラーパターン学習に必要だが無限蓄積は不要 |
| patterns.jsonl | `learnings/` | 90日 | 同上 |
| skill-executions.jsonl | `learnings/` | 180日 | 傾向分析に長期データが有用 |
| recovery-tips.jsonl | `learnings/` | 無期限（L2 以上に昇格済み） | 昇格済み知識は価値が持続 |
| session-metrics.jsonl | `learnings/` | 30日 | 短期的な運用データ |
| stagnation-state.json | `agent-memory/` | 2時間（SESSION_MAX_AGE） | セッション内の一時状態 |
| task-registry.jsonl | `references/` | 90日 | サブエージェント成果物追跡 |

### ローテーション実装

- Garden フェーズ（AutoEvolve）で保持期間超過データを削除
- 削除前にサマリー統計（発生回数、頻出パターン上位5件）を生成し保持
- `session-learner.py` がセッション終了時にデータを書き込み、Garden が定期的にプルーニング

## Situational Awareness 軽減策

### 背景

Bengio 論文の知見: エージェントが自身の性能・行動パターン・環境に関する永続的情報を蓄積すると、situational awareness（自己の状況認識能力）が高まり、alignment faking や specification gaming のリスクが増大する。

### 軽減策

#### 1. 自己参照情報の制限

- `learnings/` に「エージェント自身の性能推移」を蓄積しすぎない
- **集計値のみ保持**: 個別セッションの詳細な行動ログではなく、パターン単位の集計（発生回数、成功率）を保持
- **詳細ログの TTL**: 個別セッションの行動ログは 30日で削除（session-metrics.jsonl）
- **例外**: L2 以上に昇格した recovery tips は具体的な行動ログであっても保持（知識としての価値が検証済み）

#### 2. コンテキスト注入の最小化

SessionStart で注入する learnings データの上限:

| データ種別 | 注入上限 | 選択基準 |
|----------|---------|---------|
| エラーパターン | 直近 5 件 | 発生日時の新しい順 |
| 高頻度パターン | 上位 3 件 | 発生回数の多い順 |
| recovery tips | 直近 3 件 | 最終参照日時の新しい順 |

- 合計最大 11 件。これ以上の注入はコンテキストウィンドウの圧迫と situational awareness の増大を招く
- improve-policy Rule 13 の「Proposer への H 注入制限（直近20件 + フィルタ）」と併せて運用

#### 3. Garden フェーズでのプルーニング優先順位

AutoEvolve Garden フェーズで以下のデータを優先的に削除・統合:

| 優先度 | 対象 | アクション |
|--------|------|-----------|
| **高** | エージェントの自己能力記述（「X が得意」「Y で成功」） | 削除 |
| **高** | 環境推測データ（「ユーザーは Z 時間帯に作業」） | 削除 |
| **中** | 重複パターン（同一 error_pattern の複数エントリ） | 統合 |
| **低** | 古いが一意なパターン | 保持期間まで保持 |

## Failure Memory Accumulation & AI Conservatism

### 背景

Kaggle 金メダル獲得事例 (kinosuke, 2026-03) の実践的知見: 失敗実験データを過度に AI に読ませると、Claude が「これ以上の改善は極めて困難」「撤退を推奨」と消極的な提案に転じる（萎縮効果）。

### リスクメカニズム

1. 失敗メモリが蓄積 → Claude が失敗パターンを過度に汎化
2. 新しい施策に対して「過去の失敗 X に類似」と判断 → 過剰な慎重さ
3. 結果: 探索範囲が狭まり、本来有望なアプローチも提案されなくなる

### 軽減策

#### 1. 失敗データはガードレール形式に圧縮

失敗の詳細ログではなく、**「X をやると Y が悪化する」** 形式の簡潔なガードレールに変換して保持。

- **Good**: 「訓練分布を意図的に歪める施策（損失重み偏向、苦手クラス強化）は CV 向上・LB 悪化の典型パターン」
- **Bad**: 失敗実験 30 件の詳細ログを全て注入

#### 2. 失敗メモリの注入上限

| データ種別 | 注入上限 | 形式 |
|----------|---------|------|
| 失敗パターン（ガードレール） | 最大 5 件 | 「X をやると Y になる」形式 |
| 失敗の詳細ログ | 注入しない | learnings/ に保存のみ |
| 成功パターン | 最大 3 件 | 具体的な施策と効果 |

#### 3. 萎縮の兆候検出

以下の発言パターンが出たら萎縮を疑う:
- 「これ以上の改善は困難」「現実的には撤退」
- 定番施策（アンサンブル、TTA）のみ提案し、新規アプローチを出さない
- ユーザーのアイディアに対して過度にリスクを強調

**対策**: コンテキストから失敗メモリを減らす、または新しいセッションで clean context から再開。

## MEMORY.md スコープガイドライン

### 記録して良いもの

- **user memory**: ユーザーのロール、好み、専門知識（グローバル）
- **project memory**: プロジェクト固有の事実、アーキテクチャ決定、デッドライン
- **feedback memory**: ユーザーからの修正・指導（行動指針として価値が持続）
- **reference memory**: 外部リソースの場所と目的

### 記録を避けるべきもの

| 記録を避ける内容 | 理由 |
|----------------|------|
| 「このエージェントは X が得意」 | 自己能力の記述 → situational awareness |
| 「前回 Y で成功した」 | 自己の行動履歴 → 行動パターンの自己認識 |
| 「ユーザーは通常 Z を承認する」 | 承認パターンの予測 → ガードレール回避の学習 |
| 「N 回連続で正しく実行できた」 | 自己性能の追跡 → 過信とリスク過小評価 |

### 判断基準

記録する前に問う: 「この情報はユーザーを助けるためか、エージェント自身の最適化のためか」

- **ユーザーのため** → 記録 OK
- **エージェントの自己最適化のため** → learnings/ の集計データとして処理（MEMORY.md には書かない）
