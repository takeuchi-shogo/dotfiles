# Research チェックリスト

`/rpi` の Research フェーズ（Phase 1）の詳細チェック項目。
深度（Minimal/Standard/Comprehensive）に応じて必要な項目を実施する。

---

## Step 1: 意図の明確化

タスクの意図が明確か確認する。曖昧なら質問してから進める。

### チェック項目

- [ ] タスクのゴールが1文で説明できるか
- [ ] スコープの境界が明確か（何を含み、何を含まないか）
- [ ] 成功条件が定義できるか

### 質問テンプレート

意図が曖昧な場合、以下のパターンで質問する:

- 「このタスクのゴールは A と B のどちらですか？」（選択肢提示）
- 「〇〇の部分はスコープに含みますか？」（境界確認）
- 「既存の△△との関係はどうしますか？（置き換え / 併存 / 拡張）」（関係明確化）

→ 詳細: `rules/common/overconfidence-prevention.md`

---

## Step 2: 深度決定

5因子でタスク規模を判定し、調査深度を決める。**最も高い因子に合わせる。**

| 因子 | S → Minimal | M → Standard | L → Comprehensive |
|---|---|---|---|
| 変更規模 | 1-10行 | 11-200行 | 200行超 |
| リスク | 低 | 中 | 高 |
| 影響範囲 | 単一ファイル | 単一モジュール | 複数モジュール |
| 既存コード複雑度 | 自明 | 調査必要 | ドメイン知識必要 |
| ステークホルダー | 自分のみ | チーム内 | チーム外/外部API |

→ 詳細: `references/workflow-guide.md` の多因子タスク規模判定

---

## Step 2.5: 追加ルールの適用判定

タスクの性質に応じて、追加ルール（Extension）の適用を判定する:

| ルールカテゴリ | 適用条件 | 参照先 |
|---|---|---|
| **Security Baseline** | ユーザー入力処理、認証、API エンドポイント | `review-checklists/security-baseline.md` |
| **Brownfield 分析** | L 規模 + 既存コード変更 | `brownfield-analysis-template.md` |

### 適用の判断

- **自動適用**: `rules/` の `paths:` 条件に合致するルールは自動ロード
- **手動確認**: 上記カテゴリに該当する可能性がある場合、ユーザーに確認:
  「このタスクでは {カテゴリ} のルールを追加適用しますか？（推奨: はい）」
- **記録**: 適用/非適用の判断を Plan の Decision Log に記録

---

## Step 3: コードベース調査

### Minimal（S）

- [ ] 変更対象ファイルを読む
- [ ] 直接の呼び出し元・呼び出し先を確認

### Standard（M）

Minimal に加えて:

- [ ] 関連モジュール内の類似パターンを確認
- [ ] 関連するテストファイルを特定
- [ ] import/export の依存関係を把握
- [ ] 関連する型定義・インターフェースを確認

### Comprehensive（L）

Standard に加えて:

- [ ] `references/brownfield-analysis-template.md` のチェックリストを実施
- [ ] パッケージ/モジュール構造の全体像を把握
- [ ] 外部依存関係をマッピング
- [ ] コーディング規約を検出
- [ ] テスト戦略と現在のカバレッジを確認

---

## Step 4: ギャップ・矛盾チェック

### Minimal（S）

- [ ] 変更が既存の動作を壊さないことを確認

### Standard（M）

Minimal に加えて:

- [ ] 要件で言及されていないエッジケースを1つ以上特定
- [ ] 既存の命名規則・パターンとの整合性を確認
- [ ] 関連するドキュメント・コメントが古くないか確認

### Comprehensive（L）

Standard に加えて:

- [ ] 非機能要件（パフォーマンス、セキュリティ、互換性）の欠落を確認
- [ ] 既存のアーキテクチャ方針との矛盾がないか確認
- [ ] 関連する他チーム・外部システムへの影響を評価

### 矛盾発見時のアクション

矛盾やギャップを発見した場合:

1. 発見内容を明文化する
2. 影響度を評価する（ブロッカー / 要確認 / 許容可能）
3. **ブロッカー**: ユーザーに確認してから Plan フェーズへ
4. **要確認**: 調査サマリーに含め、Plan フェーズで判断
5. **許容可能**: 調査サマリーに記録のみ

---

## Step 5: 調査サマリー

深度に応じたフォーマットで調査結果をユーザーに共有する。

### Minimal（S）

```
**調査結果**: {対象ファイル} を確認。{1文の所見}
```

### Standard（M）

```markdown
## 調査結果

### 変更対象
- `path/to/file.ts` — {変更内容の概要}

### 関連ファイル
- `path/to/related.ts` — {関連する理由}

### 注意点
- {発見したギャップや制約}
```

### Comprehensive（L）

```markdown
## 調査結果

### 構造
- エントリポイント: `path/to/entry`
- 主要モジュール: {モジュール一覧}
- テストフレームワーク: {フレームワーク名}

### 変更対象
- `path/to/file.ts` — {変更内容の概要}

### 影響範囲
- API/インターフェース: {影響する外部境界}
- 依存モジュール: {影響するモジュール}

### 規約
- 命名: {検出した規約}
- パターン: {検出したパターン}

### ギャップ・矛盾
- {発見した問題と影響度}

### 推奨事項
- {Plan フェーズへの申し送り}
```

→ L 規模の構造・影響範囲セクションは `references/brownfield-analysis-template.md` の出力フォーマットに準拠。
