# 失敗できる意思決定リファレンス

> 出典: 曽根壮大「失敗できる意思決定とソフトウェアとの正しい歩き方」
> Plan 策定時・技術選定時に参照する意思決定フレームワーク。

---

## 判断 vs 決断

情報を集めれば理屈で答えが出せるのが **判断**、情報が不足する中で答えを出さないといけないのが **決断**。

Plan 策定時、各意思決定項目に対して問う:

| 問い | Yes → | No → |
|------|-------|------|
| 調べれば答えが出るか？ | **判断**: 調査タスクを先に実行 | **決断**: 下記の「失敗できる設計」で進む |
| やり直し可能か？ | 素早く試して検証 | 撤退条件を定め、投資上限を設ける |
| 代替案があるか？ | 最有力案で開始、失敗時にスイッチ | 慎重に検討。/spike で検証 |

**決断を迫られたら**: 「失敗しても致命傷にならない状態」を先に作る。

## Plan 前チェックリスト

M/L 規模のタスクで Plan を書く前に確認する:

- [ ] **Why**: なぜこれをやるのか（ビジネス動機・ユーザーの困りごと）
- [ ] **What**: 何を達成するのか（計測可能なゴール）
- [ ] **Who**: 誰が使う/影響を受けるのか
- [ ] **撤退条件**: いつ止めるか（時間上限、メトリクス閾値、失敗回数）
- [ ] **リスク要因**: 何がブラック・スワンになりうるか
- [ ] **先行事例**: 類似の実装・事例はないか（参照クラス予測）
- [ ] **反証**: この計画が失敗するとしたら何が原因か

全項目が埋まらなくても進めてよいが、**撤退条件** と **反証** は必須。
埋まらない項目は Open Questions として Plan に記載する。

---

## 反証探索ステップ

確証バイアスを避けるため、Plan に対して意識的に反証を探す:

1. 「この Plan が失敗するとしたら、最も可能性の高い原因は何か？」
2. 「この前提が間違っていたら、何が起きるか？」
3. 反証が見つかった場合:
   - 致命的 → Plan を修正 or 撤退
   - 許容可能 → リスクとして Plan に明記し、モニタリング方法を決める

---

## 素早く実行するための原則

- **ゆっくり考えて素早く実行する**: 計画に時間をかけ、実行は最速で。プロジェクト期間が長いほどブラック・スワンのリスクが増大する
- **初めてを減らす**: 研究（/spike, /research）とプロダクト実装を分離する。不確実性は spike で先に潰す
- **小さな成果物（インチストーン）**: WBS の各タスクを「それだけで検証可能な最小単位」に分解する。Plan のタスクリストがインチストーンになっているか確認する
