# Project Context MCP Server Template

プロジェクト固有のコンテキストを Claude Code に提供する MCP サーバーのテンプレート。

## セットアップ

### 1. テンプレートをコピー

```bash
cp -r ~/.claude/references/mcp-server-template/ ./mcp-context/
cd mcp-context
```

### 2. SUBSYSTEMS を設定

`server.py` の `SUBSYSTEMS` と `AGENTS` を編集:

```python
SUBSYSTEMS = {
    "auth": {
        "paths": ["src/auth/", "src/middleware/auth.ts"],
        "description": "認証・認可",
        "key_files": ["src/auth/provider.ts", "src/auth/types.ts"],
    },
    "api": {
        "paths": ["src/api/", "src/routes/"],
        "description": "API エンドポイント",
        "key_files": ["src/api/router.ts"],
    },
}
```

### 3. インストール & 起動

```bash
uv pip install -e .
# or
uv pip install mcp[cli]
mcp run server.py
```

### 4. Claude Code に登録

`.mcp.json` に追加:

```json
{
  "mcpServers": {
    "project-context": {
      "command": "python3",
      "args": ["mcp-context/server.py"]
    }
  }
}
```

## 提供ツール

### `list_subsystems`

- **いつ使うか**: プロジェクトの全体構造を把握したいとき。初回セッションや未知のプロジェクトで最初に呼ぶツール。特定のサブシステム名がわからないときのディスカバリーに使う。
- **入力**: なし。
- **出力**: サブシステム名・説明・対象パスの一覧（Markdown リスト形式）。`SUBSYSTEMS` が空の場合は設定を促すメッセージを返す。
- **境界条件**: サブシステム未設定時は空メッセージではなくガイダンスを返す。サブシステム数が多くてもページネーションはないため、全件返却される。

### `get_subsystem_overview`

- **いつ使うか**: 特定サブシステムのファイル構成・最近の変更を一括で把握したいとき。バグ修正や機能追加の前に対象領域の全体像を掴む用途。
- **入力**: `subsystem`（文字列、必須）— `SUBSYSTEMS` に定義済みのサブシステム名。
- **出力**: サブシステムの説明、ディレクトリ内のファイル一覧（最大50件）、直近5件の git コミットログ（Markdown 形式）。
- **境界条件**: 存在しないサブシステム名を指定すると利用可能な名前一覧を返す。ファイル数が50件を超える大規模ディレクトリでは末尾が切り捨てられる。git リポジトリ外では変更履歴セクションが空になる。

### `get_key_file`

- **いつ使うか**: サブシステムの重要ファイル（型定義、設定、エントリポイント等）の中身を直接読みたいとき。Read ツールの代わりにサブシステム文脈で読む。
- **入力**: `subsystem`（文字列、必須）、`filename`（文字列、省略可）— 省略時はキーファイル一覧を返す。
- **出力**: `filename` 指定時はファイル内容をコードブロックで返す（10,000文字超は切り捨て）。省略時は `key_files` に登録されたパス一覧。
- **境界条件**: `key_files` 未設定のサブシステムでは「未設定」メッセージを返す。大きなファイルは10,000文字で切り捨てられるため、全文が必要なら Read ツールを使う。バイナリファイルを指定するとエンコードエラーになる。

### `search_codebase`

- **いつ使うか**: プロジェクト内のコードを正規表現で横断検索したいとき。Grep ツールと異なり、サブシステムでスコープを絞れる点が利点。
- **入力**: `query`（文字列、必須）— ripgrep 互換の検索パターン。`subsystem`（文字列、省略可）— 指定するとそのサブシステムのパスに絞る。
- **出力**: マッチした行（ファイルパス:行番号:内容）を最大20件。rg が未インストールの場合は grep にフォールバック。
- **境界条件**: 結果が20件で打ち切られるため、広いパターンでは重要な結果が漏れる可能性がある。精度が必要なら Grep ツールを直接使う。rg も grep もない環境ではエラーを返す。

### `get_recent_changes`

- **いつ使うか**: 直近の変更履歴を確認したいとき。「最近何が変わった？」「このサブシステムの変更を追いたい」といった状況把握に使う。
- **入力**: `days`（整数、デフォルト7）— 遡る日数。`subsystem`（文字列、省略可）— 指定するとそのパスの変更に絞る。
- **出力**: `git log --oneline --stat` 形式の変更履歴。コミットハッシュ・メッセージ・変更ファイル統計を含む。
- **境界条件**: git リポジトリ外ではエラーを返す。`days` を大きくすると出力が膨大になる可能性がある。浅い clone（`--depth`）では古い履歴が取得できない。

### `get_agent_info`

- **いつ使うか**: プロジェクト固有のエージェント定義を確認したいとき。どのエージェントが使えるか、その役割を知りたい場合に使う。
- **入力**: `agent_name`（文字列、省略可）— 省略時は全エージェント一覧を返す。指定時はそのエージェントの詳細を返す。
- **出力**: エージェント名と役割説明（Markdown リスト形式）。
- **境界条件**: `AGENTS` が空の場合は設定を促すメッセージを返す。エージェントの実行や起動はこのツールではできない（情報参照のみ）。

### `check_context_drift`

- **いつ使うか**: ドキュメントがコードに対して古くなっていないか定期チェックしたいとき。セッション開始時やリリース前の品質確認で使う。
- **入力**: なし。
- **出力**: サブシステムごとに、コード最終更新日とドキュメント最終更新日を比較し、3日以上乖離がある場合に警告を出す。乖離がなければ「問題なし」メッセージ。
- **境界条件**: `SUBSYSTEMS` の各エントリに `doc_paths` キーが設定されていないと比較対象がなく、常に「問題なし」になる。ファイルシステムのタイムスタンプに依存するため、`git clone` 直後は全ファイルが同一タイムスタンプになり正確な検出ができない。

## カスタマイズ

- `SUBSYSTEMS`: プロジェクトのサブシステム定義。パス、説明、キーファイルを設定
- `AGENTS`: プロジェクト固有のエージェント定義
- `PROJECT_ROOT`: プロジェクトルートパス（デフォルトは server.py の親の親ディレクトリ）

## ファクトリーエージェントとの連携

`document-factory (mode: constitution)` や `document-factory` がプロジェクトセットアップ時にこのテンプレートをコピーし、プロジェクトに合わせてカスタマイズする。
