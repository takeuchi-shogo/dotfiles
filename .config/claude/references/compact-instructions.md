# Compact Instructions — コンテキスト圧縮ガイド

> PreCompact / コンテキスト圧縮時に参照する保留優先度と制約。

## Prompt Cache 安定性

LLM プロバイダは system prompt のプレフィックスをキャッシュする。キャッシュヒット時はトークンコストが大幅に下がる。

**CLAUDE.md / settings.json を変更するとキャッシュが破壊される。** セッション途中でのこれらの変更は避ける。変更が必要な場合はセッション区切りで行う。

## CC 7層メモリモデルとの位置づけ

このファイルは主に **Layer 4: Full Compaction** をハーネス側から制御する。
7層モデルの全体像は `references/cc-7-layer-memory-model.md` を参照。

## 2フェーズ出力パターン

CC の Full Compaction は2フェーズで summary を生成する:

1. **`<analysis>` ブロック** — 下書き用スクラッチパッド。モデルが思考を整理する
2. **`<summary>` ブロック** — 実際の構造化要約。context に残るのはこちらのみ

`<analysis>` は summary 品質を向上させるが、圧縮後のトークンを消費しない（strip される）。
この「思考→出力」分離は、PreCompact hook でカスタム要約を注入する際にも参考になるパターン。

## キャッシュ破壊を避けるべき操作

| 操作 | キャッシュ影響 | 代替 |
|------|-------------|------|
| セッション途中で CLAUDE.md を編集 | **破壊** | セッション区切りで変更 |
| セッション途中で settings.json を変更 | **破壊** | セッション区切りで変更 |
| Agent 定義の動的変更 | **破壊** | 変更しない。新セッションで反映 |
| ツールの動的追加/削除 | **破壊** | ToolSearch (deferred tools) で遅延ロード。セッション中のツール構成変更を避ける |
| memory/ への書き込み | **安全** | CC が ContentReplacementState で安定化 |
| tool 結果の大量出力 | **安全** | CC が Layer 1-2 で自動管理 |

## 3種の圧縮戦略

| 戦略 | コスト | 適用場面 | 現状 |
|------|--------|---------|------|
| ツール結果置換 | 極低 | 大きなツール出力 | output-offload.py で実装済み |
| LLM 要約 | 中 | 長いタスクの中間状態 | PreCompact で half-clone.sh が git 状態保存 |
| スライディングウィンドウ | 極低 | 短い対話 | Claude Code プラットフォームが管理 |

## 保留優先度（圧縮時）

圧縮時は以下の優先度で情報を保持する。上位ほど絶対に削らない:

1. **アーキテクチャ決定と理由** — 摘要不可。判断の根拠を含めて保持
2. **変更済みファイルとキー変更点** — 何をどう変えたか
3. **検証状態** — pass/fail の結論のみ（詳細ログは不要）
4. **未解決の TODO とロールバックノート** — 次のアクションに直結
5. **ツール出力** — 削除可。pass/fail の結論だけ残す

## 識別子の保全ルール

圧縮時に以下の値を絶対に改変しない:

- UUID, commit hash, PR 番号
- IP アドレス, ポート番号, URL
- ファイルパス, 関数名, 変数名
- API キー（マスク済みのもの）, エラーコード

これらを1文字でも変えると後続のツール呼び出しが失敗する。

## Agent 定義の粒度指針

> 論文 "Expert Personas Improve LLM Alignment but Damage Accuracy" (arXiv:2603.18507) の知見に基づく。
> 長いロール記述は生成品質を上げるが、知識検索精度を下げるトレードオフがある。
> 7-8B モデルでの検証結果であり、大規模モデルでは効果が小さい可能性がある（未検証）。

| タスク特性 | 推奨粒度 | 理由 |
|-----------|---------|------|
| 知識検索・事実確認（debugger, safe-git-inspector 等） | 簡潔（~50トークン） | ロール記述が事実想起を阻害しうる |
| コード生成・レビュー（code-reviewer, test-engineer 等） | 中程度（~100トークン） | ドメイン制約は有用だが冗長な修飾は不要 |
| 創造的生成・アライメント（design-reviewer, security-reviewer 等） | 詳細（~150トークン） | 詳細なロール記述が品質・安全性を向上 |

**原則**: agent 定義を書く際、「この記述はタスクの精度に貢献するか」を問う。
貢献しない修飾語・背景説明は削る。

## IFScale 閾値（指示数の上限）

> 研究データ (arXiv:2603.18507) および逆瀬川 (nyosegawa) の分析に基づく。

**150-200 指示を超えると primacy bias（最初の指示への過度な依存）が顕著化する。**

Claude Code システムプロンプト自体が ~50 指示を含むため:
- ユーザー CLAUDE.md + agent 定義 = 合計 **100 指示以内** を目標
- CLAUDE.md: `<important if>` 条件付きで必要な指示のみロード（現行設計で対応済み）
- Agent 定義: 上記「粒度指針」テーブルに従い簡潔に保つ
- 超過が疑われる場合: `/check-context` で現在の指示密度を確認

Morph の実証データ: SWE-bench でハーネス変更 22pt 変動 vs モデル交換 1pt。
**指示の質と構造 > 指示の量**。

## パッシブコンテキスト vs オンデマンドリトリーバル

> Vercel Next.js 16 実証データ (2026) に基づく。

| 方式 | パス率 | 備考 |
|------|--------|------|
| 8KB の AGENTS.md ドキュメントインデックス（パッシブコンテキスト） | **100%** | 常時ロード |
| Skills ベースのオンデマンドリトリーバル | 53% | 必要時に取得 |

パッシブコンテキスト（常時利用可能）がオンデマンドリトリーバルを **47pt** 上回る。

**含意**: CLAUDE.md / AGENTS.md は「目次」として簡潔に保ちつつ、常時ロードされるため情報密度を最大化する設計が有効。小さなインデックスファイルに高密度な指針を詰め込み、詳細は参照先に委ねる Progressive Disclosure パターンが最も高い成功率を示す。

> **永続的設計原理** (Tu 2026): スコープ分離はコンテキスト長の制限を回避するパッチではなく、**構造的必須要件**である。
> 無限コンテキストウィンドウでも信号対ノイズ比は長さに応じて低下する（認知帯域幅の劣化）。
> Progressive Disclosure は「今のモデルの制限」ではなく「情報理論的な限界」への対応。
> 詳細: `references/structured-test-time-scaling.md` §2 Mechanism II

## 圧縮 vs アーカイブ

圧縮は**有損だが追跡可能な操作**であるべき:

- 原始メッセージはアーカイブに残す（memory-archive.py が担当）
- 摘要には元のアーカイブへのパス参照を含める
- 摘要で情報が不足した場合、Agent はアーカイブを検索できる

## ファイルシステムへの移出

大量のツール出力はコンテキスト内に保持するより、ファイルに書き出して按需読み取る:

- output-offload.py: 150行超の Bash 出力を `/tmp/claude-tool-outputs/` に移出
- Agent は grep, head, tail で必要な部分だけ読み取る
- 検索結果、API レスポンス等も同様に移出を検討する
