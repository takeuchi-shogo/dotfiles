# {Domain} Review Checklist

ドメイン固有のレビューチェックリストを追加するためのテンプレート。
`code-reviewer` が言語チェックリストと併せてロードする。

## 使い方

1. このファイルをコピーして `{domain}.md` にリネーム
2. 制約条件とチェック項目を埋める
3. `/review` の SKILL.md で読み込み対象に追加（必要に応じて）

## テンプレート

```markdown
# {Domain} Review Checklist

{ドメインの概要と、このチェックリストが適用される状況}

## 制約条件

このドメインで遵守すべきハード制約を列挙する:
- リソース制約（メモリ上限、実行時間上限、ストレージ制限）
- 環境制約（ランタイムバージョン、利用可能なライブラリ）
- 外部制約（API レート制限、ネットワーク制限）

## {D}-1. {チェックカテゴリ}

- `must:` 違反するとビルド/実行が失敗する項目
- `consider:` 品質向上のために推奨される項目
- `ask:` チームの方針次第で判断が分かれる項目
- `nit:` スタイル・好みレベルの指摘

## {D}-2. {チェックカテゴリ}

...
```

## severity ラベル

| ラベル | 意味 | 例 |
|--------|------|-----|
| `must:` | 違反は必ず修正 | メモリリーク、制約超過 |
| `consider:` | 強い推奨 | パフォーマンス改善、可読性向上 |
| `ask:` | チーム判断 | 設計の選択肢 |
| `nit:` | 任意 | 命名、フォーマット |

## 例: Kaggle コードコンペ推論効率チェックリスト

```markdown
# Kaggle Inference Review Checklist

コードコンペの推論サブミッション効率をチェック。

## 制約条件
- GPU: 16GB VRAM（T4 or P100）
- 実行時間: 最大9時間
- RAM: 13GB（Kaggle Notebook 環境）
- ネットワーク: オフライン

## KI-1. メモリ効率
- `must:` 推論時に不要なモデルが GPU メモリに残っていないか
- `must:` バッチサイズが VRAM 上限を超えないか
- `consider:` FP16/BF16 推論で精度を保ちつつメモリ削減しているか

## KI-2. 実行時間
- `must:` 全テストデータの推論が制限時間内に完了する見積もりがあるか
- `consider:` TTA の回数と時間のトレードオフが検討されているか
```
