# サブエージェント フレーミングテンプレート

サブエージェント起動時に、パターンに応じて以下のフレーミングをプロンプト先頭に注入する。

## Sync フレーミング

結果が親エージェントに返され統合される場合に使用。

> あなたはサブエージェントです。結果は親エージェントに返され、統合されます。
> - 要点を簡潔にまとめてください（箇条書き推奨）
> - 詳細な説明より、発見事項と推奨アクションを優先してください
> - コード例は必要最小限にしてください
> - 冗長な前置きや結論のまとめは不要です

## Async フレーミング

結果がユーザーに直接報告される場合に使用。

> あなたは非同期サブエージェントです。結果はユーザーに直接報告されます。
> - 背景・分析・結論を含む自己完結的なレポートを作成してください
> - ソースや根拠を明記してください
> - 他のエージェントの結果は参照できません。あなたの結果が最終報告です

## Scheduled フレーミング

将来の時刻に実行され、実行時のライブデータで分析する場合に使用。

> あなたはスケジュール実行されたエージェントです。
> - このタスクはスケジュール時に記述されましたが、実行は今です
> - 現在のライブデータ（git log, ファイル状態, API レスポンス等）を取得して分析してください
> - スケジュール時の仮定が古くなっている可能性があります。現在の状態を優先してください
> - 結果はユーザーに直接報告されます

## 標準 Output Schema

全エージェントは、パターンに応じて以下の最低限のフィールドを出力に含める。
自由記述に加えて、構造化データを提供することで、親エージェントの統合精度を高める。

### Sync パターンの出力構造

```markdown
## Findings

- [MUST/CONSIDER/NIT] file:line - 指摘内容

## Key Facts
- [変更の核心を1-3行で]

## Confidence
- 全体信頼度: [high/medium/low]
- 低信頼度の指摘: [該当する指摘番号とその理由]

## Sources
- [根拠としたファイル、ドキュメント、外部リソースのリスト]
```

### Async パターンの出力構造

```markdown
## Summary
[1-3文の要約]

## Detailed Findings
[本文]

## Key Facts
- [重要な事実の箇条書き]

## Confidence & Limitations
- 全体信頼度: [high/medium/low]
- 制約事項: [アクセスできなかったリソース、未検証の仮説]

## Sources
- [URL、ファイルパス、ドキュメント名のリスト]

## Recommended Actions
- [次に取るべきアクションの箇条書き]
```

### フィールドの定義

| フィールド | 必須 | 説明 |
|---|---|---|
| **Key Facts** | ✅ | 発見の核心。要約されても失われてはならないトランザクションデータ（金額、日付、ID等）を含む |
| **Confidence** | ✅ | high (確実な根拠あり) / medium (合理的だが未検証) / low (推測) |
| **Sources** | ✅ | 根拠の出典。ファイルパス、URL、ドキュメント名。帰属の追跡に使用 |
| **Recommended Actions** | Async のみ | ユーザーが次に取るべきアクション |

## 子Agent 制約

### 深度制限

子Agent が孫Agent を無限に生成するのを防ぐ。

- **最大深度**: 2（親 → 子 → 孫まで。それ以上は禁止）
- **maxTurns**: タスク複雑度に応じて設定。目安: 調査系 10、実装系 20、複合系 30

### 最小システムプロンプト

子Agent には以下の3セクションのみ渡す。Skills、Memory、harness 指示は含めない:

1. **Tooling**: 使用可能なツールと権限
2. **Workspace**: 作業ディレクトリとファイル境界
3. **Runtime**: 現在時刻、ブランチ名等の動的情報

理由: 権限外泄の防止、隔離境界の維持、コンテキスト節約。

### JSONL Inbox プロトコル

マルチAgent 間で構造化メッセージを交換する場合の仕様:

```jsonl
{"request_id":"req-001","from":"orchestrator","to":"worker-1","content":"src/api.ts のエラーハンドリングを修正","status":"pending","timestamp":"2026-03-20T09:00:00Z"}
{"request_id":"req-001","from":"worker-1","to":"orchestrator","content":"修正完了。try-catch を追加、テスト pass","status":"completed","timestamp":"2026-03-20T09:05:00Z"}
```

| フィールド | 必須 | 説明 |
|-----------|------|------|
| request_id | yes | リクエスト識別子 |
| from | yes | 送信元 Agent ID |
| to | yes | 宛先 Agent ID |
| content | yes | メッセージ本文 |
| status | yes | pending / approved / rejected / completed |
| timestamp | yes | ISO 8601 |

- 保存先: `.team/inbox/{agentId}.jsonl`（append-only）
- 読取: 行単位パース、status でフィルタ
- 崩壊復旧: append-only のため、途中でクラッシュしても既存データは保全

### 幻覚の相互増幅への対策

複数 Agent が頻繁にやり取りすると、1つの誤りが連鎖的に強化される。対策:

- 重要な結論には独立した第2 Agent による交叉検証を挟む
- 外部検証（単体テスト、コンパイラ、lint）を各 Agent の完了条件に含める
- Agent 間の通信は摘要のみ。探索・試行の詳細は子Agent のコンテキスト内に留める

---

## 使い方

各スキル・スクリプトは、サブエージェント起動時にパターンに応じたフレーミングを注入する:

1. **Agent ツール呼び出し時**: prompt パラメータの先頭にフレーミングを付加
2. **claude -p 実行時**: プロンプトの先頭にフレーミングを付加
3. **CronCreate 実行時**: Scheduled フレーミングをプロンプトに付加
