# Staleness Criteria — 鮮度判定基準

> **存在 ≠ 十分** (`rules/common/existence-vs-sufficiency.md`):
> 鮮度（更新日）だけでなく、内容の十分性も評価する。
> ファイルが存在し最近更新されていても、内容がコードと乖離していれば不健全。

## Content Sufficiency（内容の十分性）

鮮度チェックに加え、以下の観点でドキュメントの十分性を評価する:

| 観点 | OK | WARNING |
|------|-----|---------|
| **カバレッジ** | 対象コードの主要機能を説明 | テンプレートの見出しだけ埋まっている |
| **具体性** | 具体的なコマンド・手順・例を含む | 「適切に処理する」等の抽象表現が多い |
| **コード整合** | 記載のパス・関数名・設定値が現在のコードと一致 | 存在しないファイルやリネーム済みの関数を参照 |

## Document Staleness

| 閾値 | 判定 | アクション |
|------|------|-----------|
| 0-30日 | OK | なし |
| 31-60日 | STALE | レビュー推奨 |
| 61日以上 | CRITICAL | 即座に更新 or アーカイブ |

## Code-Document Drift

| 条件 | 判定 |
|------|------|
| コード変更後7日以内にドキュメント更新 | OK |
| コード変更後7-14日 | WARNING |
| コード変更後14日以上 | DRIFT |

## Skill Benchmark Staleness

| 閾値 | 判定 |
|------|------|
| 0-30日 | OK |
| 31-60日 | STALE → `/skill-audit` 推奨 |
| 未実施 | NEVER → `/skill-audit` 必須 |

## Reference Integrity

参照先ファイルが存在しない場合は **即 BROKEN** と判定。
ファイルが移動された場合は **MOVED** と判定し、新パスを提示。
