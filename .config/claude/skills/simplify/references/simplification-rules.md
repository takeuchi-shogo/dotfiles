# Simplification Detection Rules

## 1. DRY 違反検出

| ルール | しきい値 | 説明 |
|--------|---------|------|
| 同一コードブロック | 2+ 箇所 | 5行以上の同一コードが2箇所以上 |
| 類似パターン | 3+ 箇所 | 構造が同じで値だけ異なるパターンが3箇所以上 |
| コピペ関数 | 2+ 箇所 | 80%以上同一の関数が2つ以上 |

**除外**: テストコード内の類似パターン（テストの可読性優先）

## 2. Nesting 検出

| ルール | しきい値 | 説明 |
|--------|---------|------|
| 条件分岐の深さ | > 3 | if/switch のネストが3段超 |
| コールバックネスト | > 2 | コールバック/Promise チェーンが2段超 |
| ループ内条件 | > 2 | ループ内の条件分岐が2段超 |

**緩和策**: Early return、Guard clause、関数分割

## 3. Complexity 検出

| ルール | しきい値 | 説明 |
|--------|---------|------|
| 関数の行数 | > 50 | 単一関数が50行超 |
| パラメータ数 | > 4 | 関数の引数が4個超 |
| 分岐数 | > 10 | 単一関数の条件分岐が10超 |
| God function 化 | CC>10 の関数への分岐追加 | 既に複雑な関数にさらにロジックを追加するパターン |

### God Function 化パターン（LLM 固有）

LLM は反復的編集で新ロジックを既存関数にパッチする傾向がある（SlopCodeBench, 2026）。
以下のシグナルを検出したら `must:` で分割を提案する:

| シグナル | 具体例 | 推奨アクション |
|---------|--------|---------------|
| elif/case チェーンの成長 | `if cmd == 'a': ... elif cmd == 'b': ... elif cmd == 'c': ...` | dispatch table or command パターンに抽出 |
| 同一関数内の scaffold コピペ | 同じ引数パース構造が複数分岐に出現 | 共通ヘルパーに抽出 |
| 関数内の独立ブロック | コメントで区切られた独立セクション | 各ブロックを個別関数に抽出 |
| 条件分岐内の長い実装 | if ブロック内が20行超 | ブロック本体を別関数に抽出 |

## 4. Reuse 候補検出

| ルール | 条件 | 説明 |
|--------|------|------|
| ユーティリティ候補 | 2+ 箇所で同一変換 | 同じデータ変換が複数箇所で出現 |
| 共通バリデーション | 3+ 箇所で同一検証 | 同じバリデーションロジックが繰り返し |
| 設定値ハードコード | 2+ 箇所で同一マジックナンバー | 定数化すべき値 |

## 5. Query 検出

| ルール | パターン | 説明 |
|--------|---------|------|
| N+1 問題 | ループ内 DB/API 呼び出し | バッチ化を提案 |
| 未使用フィールド取得 | SELECT * 相当 | 必要フィールドのみ取得を提案 |
| 繰り返し同一クエリ | 同一条件の複数回実行 | キャッシュまたは結果の再利用を提案 |

## 6. LLM 冗長パターン検出

LLM が生成しやすい冗長コードパターン（SlopCodeBench, 2026 の 137 AST-Grep ルールから主要なものを抽出）。

| ルール | パターン | 説明 |
|--------|---------|------|
| Identity 変換 | `[x for x in items if cond]` を filter 相当で使用（変換なし） | filter / 直接条件で代替可能 |
| Single-use 変数 | 意味のない名前の中間変数が直後に1回だけ使用 | インライン化を提案 |
| 空チェック + continue | `if not cond: continue` を loop 冒頭に配置 | filter で loop 対象を絞る方が明瞭な場合あり |
| 構造的クローン | 値だけ異なる同構造コードが 3+ 箇所 | パラメータ化した共通関数に抽出 |
| 防御的過剰チェック | 到達不能な条件への defensive check | フレームワーク保証を信頼。境界でのみ検証 |

**注意**: これらは「必ず修正」ではなく「検出して判断」。可読性が上がる場合にのみ修正を提案する。

## 言語別の追加ルール

各言語の review checklist（`references/review-checklists/`）が存在する場合、そのルールも適用する。
