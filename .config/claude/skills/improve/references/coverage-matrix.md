# Coverage Matrix — 分析の網羅性保証

> Phase 2a (meta-analyzer) はこのマトリックスの全問いに回答する義務がある。
> `ANSWERED` / `INSUFFICIENT_DATA` / `NOT_APPLICABLE` のいずれかを明示すること。
> 「問題なし」と「データ不足」を絶対に混同しない。

## Status 定義

| Status | 意味 | 要件 |
|--------|------|------|
| `ANSWERED` | データに基づいて回答済み | evidence に具体的なデータ参照（件数, session_id, ファイル名）を含む |
| `INSUFFICIENT_DATA` | データ不足で回答不能 | 何が足りないかを明記（例: errors.jsonl が 2 件のみ） |
| `NOT_APPLICABLE` | このカテゴリが対象外 | 理由を明記（例: スキル実行データが 0 件） |

## errors カテゴリ

| # | 必須問い | 回答基準 |
|---|---------|---------|
| E1 | **繰り返しエラー**: 3回以上の同一エラーパターンはあるか？ | error_pattern でグルーピング、件数付き |
| E2 | **根本原因チェーン**: 即時原因→近因→根本原因を特定できたか？ | 最低 1 つのエラーで 3 層分析を実施 |
| E3 | **トレンド**: 直近 10 セッション vs 前 10 セッションで増減は？ | 増加/減少/横ばい + 変化率 |
| E4 | **予防可能性**: hook/rule で機械的に防げるエラーはあるか？ | 具体的な hook/rule の提案付き |
| E5 | **環境 vs コード**: エラーの原因は環境固有か、コード固有か？ | 分類比率を報告 |

## quality カテゴリ

| # | 必須問い | 回答基準 |
|---|---------|---------|
| Q1 | **上位違反**: 頻度上位 5 ルールとその件数は？ | ルール名 + 件数 + トレンド |
| Q2 | **新規 vs 再発**: 新たに出現した違反と既知の再発の比率は？ | 比率 + 新規違反のリスト |
| Q3 | **エラーとの共起**: 品質違反とエラーが同一セッションで共起するパターンは？ | 共起ペアと頻度 |
| Q4 | **改善効果**: 前回 /improve で対策した違反は減少したか？ | 対策済み項目のビフォーアフター |

## skills カテゴリ

| # | 必須問い | 回答基準 |
|---|---------|---------|
| S1 | **健全性マップ**: Failing / Degraded / Healthy の分布は？ | 各スキルの status + score + trend |
| S2 | **失敗根本原因**: Failing スキルに共通する根本原因は？ | パターン分類（トリガー不足, instruction 不備, 環境変化 等） |
| S3 | **使用頻度 vs 品質**: 頻繁に使われるのに品質が低いスキルは？ | 使用回数 × score のマトリックス |
| S4 | **未使用スキル**: 30日以上実行されていないスキルは？ | リスト + retire 候補の判定 |

## agents カテゴリ

| # | 必須問い | 回答基準 |
|---|---------|---------|
| A1 | **効率外れ値**: 平均ターン数が全体の 1.5σ 以上のエージェントは？ | エージェント名 + 平均ターン数 + 比較値 |
| A2 | **Knowledge Embedding Ratio**: ドメイン知識比率 < 50% のエージェントは？ | エージェント名 + 概算比率 |
| A3 | **ルーティング精度**: 誤って起動された / 起動されるべきだったエージェントは？ | 具体的なケースと根拠 |

## environment カテゴリ

| # | 必須問い | 回答基準 |
|---|---------|---------|
| V1 | **アクセス不能な情報**: エージェントが必要だがアクセスできない情報は？ | 具体例 + 繰り返し同じ調査をしている証拠 |
| V2 | **欠落フィードバックループ**: ミスが伝播してから検出されるケースは？ | 検出が遅れた具体例 + 早期検出可能なポイント |
| V3 | **コンテキスト汚染**: 無関係な情報でコンテキストが汚染されているか？ | 具体例（大量出力, 古い情報の残留 等） |
| V4 | **機械的強制**: エージェント判断に依存しているが hook で強制すべき制約は？ | 具体的な hook 提案 |

## 起動条件

| カテゴリ | 最低データ要件 |
|---------|-------------|
| errors | errors.jsonl 3 件以上 |
| quality | quality.jsonl 3 件以上 |
| skills | skill-executions.jsonl 5 件以上 |
| agents | session-metrics.jsonl 5 件以上 |
| environment | errors.jsonl 20 件以上 AND 前回分析から 7 日以上 |

データ要件を満たさないカテゴリは全問い `INSUFFICIENT_DATA` で埋める。
「データがないから分析しない」ではなく「データが不足している」と明示する。

## Codex Deep Analysis (Phase 2b) への入力

Phase 2a の完成した Coverage Matrix を Codex に渡す。
Codex は以下の観点で Matrix を検証・補強する:

1. **ANSWERED 項目の深度チェック**: 根本原因が浅い分析を指摘
2. **見落としパターン**: Matrix に含まれない重要な観点の追加
3. **クロスカテゴリ構造**: 複数カテゴリにまたがる構造的問題の検出
4. **分析品質レーティング**: 各カテゴリを THOROUGH / ADEQUATE / SHALLOW で判定

SHALLOW 判定のカテゴリは Phase 3 に進む前に追加分析を行う。
