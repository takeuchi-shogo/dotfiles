# API 設計詳細ガイド

## ページネーション戦略

### Cursor-based（推奨）
- 大規模データセットに最適
- ページの挿入/削除に強い
- `?cursor=eyJpZCI6MTIzfQ&limit=20`

### Offset-based
- シンプルだが大量データで性能劣化
- `?page=3&per_page=20`
- 管理画面等の小規模データ向け

## レート制限パターン

### Token Bucket
- バースト許容 + 平均レート制限
- ヘッダーで残量を通知: `X-RateLimit-Remaining`, `X-RateLimit-Reset`

### Sliding Window
- より正確なレート計算
- Redis の ZSET で実装可能

## データベースマイグレーション

### 安全な手順
1. **新カラム追加**: nullable で追加 → デプロイ → データ埋め → NOT NULL に変更
2. **カラム削除**: コードから参照を削除 → デプロイ → カラム削除
3. **テーブルリネーム**: 新テーブル作成 → 二重書き込み → 切り替え → 旧テーブル削除

### アンチパターン
- ロック時間の長い ALTER TABLE を本番で直接実行
- データマイグレーションとスキーママイグレーションの混在
- ロールバック不可能なマイグレーション

## エラーハンドリング設計

### HTTP ステータスコードの使い分け
- 400: クライアントの入力エラー（バリデーション失敗）
- 401: 認証が必要（トークン無効/期限切れ）
- 403: 認証済みだが権限不足
- 404: リソースが存在しない
- 409: 競合（楽観的ロック失敗等）
- 422: 構文は正しいが意味的に処理不可
- 429: レート制限超過
- 500: サーバー内部エラー（クライアントに詳細を漏らさない）
