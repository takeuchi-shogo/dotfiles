# アーキテクチャパターン詳細

## イベント駆動アーキテクチャ

### いつ使う
- サービス間の疎結合が必要
- 非同期処理が多い
- イベントの監査ログが必要

### いつ使わない
- 強い整合性が必要（銀行の送金等）
- シンプルなリクエスト/レスポンスで十分
- デバッグの容易さを優先する初期フェーズ

### パターン
- **Event Notification**: 最小限のイベントを発行し、受信側が必要なデータを取得
- **Event-Carried State Transfer**: イベントにデータを含め、受信側がローカルコピーを保持
- **Event Sourcing**: 状態の変更をすべてイベントとして記録

## CQRS (Command Query Responsibility Segregation)

### いつ使う
- 読み取りと書き込みのスケーリング要件が大きく異なる
- 読み取りモデルと書き込みモデルが複雑に乖離
- イベントソーシングと組み合わせる場合

### いつ使わない
- 単純なCRUDアプリケーション
- 読み書きの負荷が均一
- チームがパターンに不慣れ

## Saga パターン

### いつ使う
- マイクロサービス間の分散トランザクション
- 各サービスが独自のDBを持つ

### 2つのアプローチ
- **Choreography**: 各サービスがイベントを発行し次のステップをトリガー（シンプルなフロー向け）
- **Orchestration**: 中央のオーケストレーターがフロー全体を管理（複雑なフロー向け）

### 補償トランザクション
失敗時に前のステップを巻き戻す補償アクションを必ず定義する。

## API Gateway / BFF

### API Gateway
- 全クライアント共通のエントリポイント
- 認証、レート制限、ルーティングを集約
- 用途: 公開API、統一エンドポイント

### BFF (Backend for Frontend)
- クライアント種別ごとに専用のバックエンド
- Web / Mobile / 3rd Party で異なるデータ加工
- 用途: クライアントごとに最適化されたAPI

## Circuit Breaker

### 3つの状態
1. **Closed**: 正常動作、リクエストを通過
2. **Open**: 障害検出、リクエストを即座に拒否（フォールバック返却）
3. **Half-Open**: 一部リクエストを試行し、回復を確認

### 設定パラメータ
- failure threshold: Open に遷移するまでの失敗数
- timeout: Open から Half-Open に遷移するまでの待機時間
- success threshold: Half-Open から Closed に戻るまでの成功数
