# Senior Architect — 設計意思決定ガイド

システムアーキテクチャに関する判断を支援するガイド。実装詳細ではなく「何を選ぶか」に特化。

## 最初に: 診断フロー（処方箋の前に診断）

アーキテクチャの選択に入る前に、**問題の根本原因を特定する**。

```
ユーザーの訴え → 本当にアーキテクチャの問題か？

「デプロイが遅い」    → CI/CDパイプラインの問題では？ テストの問題では？
「変更の影響範囲が広い」→ モジュール境界の問題では？（モノリスのまま修正可能）
「スケールしない」    → DB クエリ最適化で解決しないか？ キャッシュで十分では？
「チーム間のコンフリクト」→ コードオーナーシップの問題では？ ブランチ戦略で解決しないか？
```

**診断ステップ:**
1. **症状を列挙する** — ユーザーが感じている具体的な痛み
2. **根本原因仮説を立てる** — アーキテクチャ起因 vs プロセス起因 vs 技術的負債
3. **最小コスト検証** — 仮説を確認する最も安い方法を提案
4. **確認後に処方箋** — 原因が確定してから選択肢を提示

## デシジョンツリー

### 1. モノリス vs マイクロサービス

```
チーム規模 ≤ 5人        → モノリス（モジュラー）
チーム規模 6-15人       → モジュラーモノリス + 段階的分離
チーム規模 > 15人       → マイクロサービス検討
スタートアップ初期      → 常にモノリスから始める
```

| 判断基準 | モノリス推奨 | マイクロサービス推奨 |
|----------|-------------|---------------------|
| チーム構成 | 単一チーム | 複数の自律チーム |
| データ整合性 | 強い整合性要件 | 結果整合性で十分 |
| デプロイ | 一括リリースOK | 独立デプロイ必須 |
| スケーリング | 均一な負荷 | コンポーネント別に異なる |
| フェーズ | MVP・初期 | 成熟・拡張期 |

### 2. 同期 vs 非同期通信

| 条件 | 推奨パターン |
|------|-------------|
| レスポンスが即座に必要 | 同期（REST / gRPC） |
| 結果整合性で十分 | 非同期（イベント / メッセージキュー） |
| 高スループット要件 | 非同期 |
| トランザクション境界 | 同期 + Saga パターンで補完 |
| Fire-and-forget | 非同期（キュー） |

### 3. API スタイル選定

| 条件 | REST | GraphQL | gRPC |
|------|------|---------|------|
| 公開API | ◎ | ○ | × |
| モバイルクライアント | ○ | ◎ | ○ |
| 内部サービス間 | ○ | × | ◎ |
| リアルタイム | WebSocket | Subscription | Stream |
| 単純なCRUD | ◎ | 過剰 | 過剰 |

### 4. 各選択の反対論と実装コスト

判断を下す際は、**推奨する選択肢だけでなく、なぜ他を選ばないかも説明する**。

| 選択 | なぜ選ぶか | なぜ選ばないか | 実装コスト目安 |
|------|-----------|---------------|---------------|
| モノリス維持 | シンプル、チーム小、整合性重要 | デプロイ独立性なし、部分スケール不可 | 0（現状維持） |
| モジュラーモノリス | 段階的移行可、境界明確化 | 本当の独立デプロイではない | 1-2週（境界整理） |
| マイクロサービス | 独立スケール、独立デプロイ | 分散システムの複雑性、運用コスト増大 | 2-6ヶ月（基盤構築含む） |
| REST API | 標準的、キャッシュ容易、ツール豊富 | Over/Under-fetching | 数日 |
| GraphQL | 柔軟なクエリ、モバイル最適 | 学習コスト、キャッシュ複雑 | 1-2週 |
| gRPC | 高性能、型安全 | ブラウザ直接不可、デバッグ困難 | 1-2週 |
| 同期通信 | 実装シンプル、即座に結果 | 可用性低下（N^乗の法則）、カスケード障害 | 数日 |
| 非同期通信 | 疎結合、高スループット | 結果整合性、デバッグ困難 | 1-3週（基盤含む） |

**具体的技術推奨:**
- メッセージキュー: MVP → Redis Streams / SQS、スケール → Kafka / NATS
- サービスメッシュ: Istio（K8s）/ Linkerd（軽量）
- API Gateway: Kong / AWS API Gateway / Traefik
- 監視: Datadog / Grafana + Prometheus / AWS CloudWatch
- 分散トレーシング: OpenTelemetry + Jaeger / Zipkin

## スケーラビリティチェックリスト

設計レビュー時に以下を確認:

- [ ] **ステートレス設計**: セッションは外部ストア（Redis等）に保存
- [ ] **水平スケーリング**: ロードバランサー配下で複数インスタンス起動可能
- [ ] **ボトルネック特定**: DB / API / ネットワーク / 計算のどこが律速か
- [ ] **キャッシュ戦略**: CDN（静的）/ Redis（アプリ）/ in-memory（ホットデータ）
- [ ] **DB戦略**: リードレプリカ / コネクションプール / 必要ならシャーディング
- [ ] **障害分離**: Circuit Breaker / Bulkhead / Retry with backoff
- [ ] **非同期化**: 重い処理はキューに逃がす（メール送信、画像処理等）
- [ ] **監視**: メトリクス / ログ / トレーシング（Observability の3本柱）

## アーキテクチャ図の生成

Mermaid 記法で以下の C4 Model 図を生成する:

1. **System Context 図** (Level 1): システム全体と外部アクター・外部システムの関係
2. **Container 図** (Level 2): アプリケーション、DB、メッセージキュー等の構成要素
3. **Component 図** (Level 3): コンテナ内部のモジュール構成

データフローは sequence diagram で可視化。

## 技術選定フレームワーク

新しい技術を導入する前に評価:

| 評価軸 | 質問 |
|--------|------|
| **必要性** | 既存技術で解決できないか？ |
| **成熟度** | プロダクション実績は十分か？ |
| **チーム** | 学習コストは許容範囲か？ |
| **運用** | 監視・デバッグ・障害対応は容易か？ |
| **エコシステム** | ライブラリ・ツール・コミュニティは活発か？ |

**判断に迷う場合**: 小規模 PoC で検証してから決定。

## 出力の品質基準

アーキテクチャ提案では以下を必ず含める:

1. **診断**: 問題の根本原因と、アーキテクチャ変更が本当に必要な理由
2. **推奨 + 反対論**: なぜこの選択肢か、なぜ他ではないか
3. **実装コスト**: フェーズごとの時間見積と必要リソース
4. **リスクと軽減策**: 選択に伴うリスクと対処法
5. **明日のアクション**: 最初に着手すべき具体的ステップ

詳細なパターン集は `references/architecture-patterns.md` を参照。
