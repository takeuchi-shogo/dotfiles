# Senior Architect — 設計意思決定ガイド

システムアーキテクチャに関する判断を支援するガイド。実装詳細ではなく「何を選ぶか」に特化。

## デシジョンツリー

### 1. モノリス vs マイクロサービス

```
チーム規模 ≤ 5人        → モノリス（モジュラー）
チーム規模 6-15人       → モジュラーモノリス + 段階的分離
チーム規模 > 15人       → マイクロサービス検討
スタートアップ初期      → 常にモノリスから始める
```

| 判断基準 | モノリス推奨 | マイクロサービス推奨 |
|----------|-------------|---------------------|
| チーム構成 | 単一チーム | 複数の自律チーム |
| データ整合性 | 強い整合性要件 | 結果整合性で十分 |
| デプロイ | 一括リリースOK | 独立デプロイ必須 |
| スケーリング | 均一な負荷 | コンポーネント別に異なる |
| フェーズ | MVP・初期 | 成熟・拡張期 |

### 2. 同期 vs 非同期通信

| 条件 | 推奨パターン |
|------|-------------|
| レスポンスが即座に必要 | 同期（REST / gRPC） |
| 結果整合性で十分 | 非同期（イベント / メッセージキュー） |
| 高スループット要件 | 非同期 |
| トランザクション境界 | 同期 + Saga パターンで補完 |
| Fire-and-forget | 非同期（キュー） |

### 3. API スタイル選定

| 条件 | REST | GraphQL | gRPC |
|------|------|---------|------|
| 公開API | ◎ | ○ | × |
| モバイルクライアント | ○ | ◎ | ○ |
| 内部サービス間 | ○ | × | ◎ |
| リアルタイム | WebSocket | Subscription | Stream |
| 単純なCRUD | ◎ | 過剰 | 過剰 |

## スケーラビリティチェックリスト

設計レビュー時に以下を確認:

- [ ] **ステートレス設計**: セッションは外部ストア（Redis等）に保存
- [ ] **水平スケーリング**: ロードバランサー配下で複数インスタンス起動可能
- [ ] **ボトルネック特定**: DB / API / ネットワーク / 計算のどこが律速か
- [ ] **キャッシュ戦略**: CDN（静的）/ Redis（アプリ）/ in-memory（ホットデータ）
- [ ] **DB戦略**: リードレプリカ / コネクションプール / 必要ならシャーディング
- [ ] **障害分離**: Circuit Breaker / Bulkhead / Retry with backoff
- [ ] **非同期化**: 重い処理はキューに逃がす（メール送信、画像処理等）
- [ ] **監視**: メトリクス / ログ / トレーシング（Observability の3本柱）

## アーキテクチャ図の生成

Mermaid 記法で以下の C4 Model 図を生成する:

1. **System Context 図** (Level 1): システム全体と外部アクター・外部システムの関係
2. **Container 図** (Level 2): アプリケーション、DB、メッセージキュー等の構成要素
3. **Component 図** (Level 3): コンテナ内部のモジュール構成

データフローは sequence diagram で可視化。

## 技術選定フレームワーク

新しい技術を導入する前に評価:

| 評価軸 | 質問 |
|--------|------|
| **必要性** | 既存技術で解決できないか？ |
| **成熟度** | プロダクション実績は十分か？ |
| **チーム** | 学習コストは許容範囲か？ |
| **運用** | 監視・デバッグ・障害対応は容易か？ |
| **エコシステム** | ライブラリ・ツール・コミュニティは活発か？ |

**判断に迷う場合**: 小規模 PoC で検証してから決定。

詳細なパターン集は `references/architecture-patterns.md` を参照。
