# インターフェース設計 (Interface Design)

選ばれた深化候補に対して、代替インターフェースを探りたいときの並列サブエージェントパターン。Ousterhout の "Design It Twice" — 最初のアイデアが最良である可能性は低い。

[LANGUAGE.md](LANGUAGE.md) の語彙 (モジュール / インターフェース / シーム / アダプター / レバレッジ) を使う。

## 起動条件

**デフォルト**: SKILL.md の Step 4 grilling loop 内で 1 案を設計する。このサブスキルは起動しない。

**並列起動するのは**: ユーザーが「複数案を見たい」と明示要求した時のみ。トークンコスト (各 brief ~500 tokens × 3-4 = ~1500-2000 tokens) を踏まえて慎重に。

## Process

### 1. 問題空間を枠付ける

サブエージェント起動前に、選ばれた候補に対するユーザー向けの問題空間説明を書く:

- 新インターフェースが満たすべき制約
- 依存とそのカテゴリ ([DEEPENING.md](DEEPENING.md))
- 制約を具体化するための大まかなコードスケッチ — 提案ではなく、制約を具体にするだけ

ユーザーに見せ、すぐ Step 2 に進む。ユーザーは読み考える間にサブエージェントが並列で動く。

### 2. サブエージェント並列起動

Agent ツールで 3 つを並列起動 (Ports & Adapters 該当時のみ 4 つ目を追加)。それぞれが **根本的に異なる** インターフェースを生む。

各サブエージェントへ独立した技術ブリーフ (file paths, 結合詳細, [DEEPENING.md](DEEPENING.md) の依存カテゴリ, シーム後ろにあるもの) を与える。Step 1 のユーザー向け説明とは独立。各エージェントに異なる設計制約を与える:

- Agent 1: 「インターフェースを最小化 — エントリポイント 1-3 個。エントリポイント当たりのレバレッジ最大化」
- Agent 2: 「柔軟性最大化 — 多用途と拡張をサポート」
- Agent 3: 「最頻呼び出し側に最適化 — デフォルトケースを自明に」
- Agent 4 (該当時): 「シーム越し依存に対する Ports & Adapters 周りで設計」

ブリーフに [LANGUAGE.md](LANGUAGE.md) 語彙と `docs/glossary.md` (存在すれば) のドメイン語彙を含めて、各サブエージェントが命名で一貫するように。

各サブエージェント出力:

1. インターフェース (型、メソッド、引数 — 加えて不変条件、順序、エラーモード)
2. 呼び出し側がどう使うか示すサンプル
3. 実装がシーム後ろに何を隠すか
4. 依存戦略とアダプター ([DEEPENING.md](DEEPENING.md))
5. トレードオフ — レバレッジが高いところ、薄いところ

### 3. 提示と比較

設計を順に提示し (ユーザーが各案を吸収できるよう)、その後散文で比較する。**深さ** (インターフェースでのレバレッジ)、**局所性** (変更が集まる場所)、**シーム配置** で対比する。

比較後、自分の推薦を出す: どれが最強か、その理由。複数案の要素を組み合わせるとよさそうならハイブリッドを提案。意見を持つ — ユーザーはメニューではなく強い読みを求めている。
