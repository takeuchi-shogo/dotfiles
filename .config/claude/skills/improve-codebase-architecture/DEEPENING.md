# 深化 (Deepening)

浅いモジュール群を依存関係を踏まえて安全に深化する方法。
[LANGUAGE.md](LANGUAGE.md) の語彙 (モジュール / インターフェース / シーム / アダプター) を前提とする。

## 依存カテゴリ

候補を深化する時、依存を分類する。カテゴリが「シーム越しのテスト方法」を決める。

### 1. インプロセス (In-process)

純粋計算、メモリ内状態、I/O なし。常に深化可能 — モジュールをマージし、新しいインターフェースで直接テスト。アダプター不要。

### 2. ローカル代替可能 (Local-substitutable)

ローカルでテスト用の代用が存在する依存 (Postgres → PGLite、ファイルシステム → in-memory)。代用が存在すれば深化可能。深化したモジュールはテストスイート内で代用を走らせて検証する。シームは内部、外部インターフェースにポートを露出しない。

### 3. リモート + 自前 (Ports & Adapters)

ネットワーク越しの自前サービス (マイクロサービス、内部 API)。シームに **ポート (interface)** を定義する。深いモジュールがロジックを持ち、トランスポートは **アダプター** として注入される。テストは in-memory アダプター、本番は HTTP / gRPC / queue アダプター。

提案の形: *「シームにポートを定義し、本番用に HTTP アダプター、テスト用に in-memory アダプターを実装する。ロジックは 1 つの深いモジュールに集める、ネットワーク越しに配置されていても」*

### 4. 真の外部 (Mock)

自分の管理外のサードパーティ (Stripe、Twilio 等)。深化したモジュールは外部依存を注入されたポートとして取り、テストはモックアダプターを提供する。

## シーム規律

- **アダプター 1 個 = 仮説のシーム、2 個 = 本物のシーム**: 少なくとも 2 つのアダプターが正当化されない限りポートを導入しない (典型的には本番 + テスト)。1 アダプターのシームはただの間接化。
- **内部シーム vs 外部シーム**: 深いモジュールは内部シーム (実装内部、自分のテスト用) と外部シーム (インターフェース上) の両方を持てる。テストが使うからといって内部シームをインターフェースに露出するな。

## テスト戦略: 置き換える、重ねるな

- 浅いモジュールへの古いユニットテストは、深化したモジュールのインターフェースでテストが書かれた時点で **無駄** — 削除する。
- 新しいテストは深化したモジュールのインターフェースで書く。**インターフェース = テスト面**。
- テストはインターフェース越しの観測可能な結果を主張する、内部状態ではない。
- テストは内部リファクタを生き残るべき — 挙動を記述するもので実装を記述するものではない。実装を変えた時テストが変わるなら、それはインターフェースの **奥** をテストしている。

## Agent-Native との関係

[../../references/agent-native-code-design.md](../../references/agent-native-code-design.md) の 5 原則は、深化の **副作用** として自然に達成される:

- **原則 1 (Grep-able 命名)**: 深化したモジュール = 1 箇所に集まる = grep 可能
- **原則 2 (Collocated Tests)**: インターフェース面でのテスト = ソース隣接
- **原則 3 (機能単位)**: 深化 = 機能単位の凝集
- **原則 5 (API 境界)**: シーム規律 = API 境界の明確化

深化は agent-native の **どう実現するか** を提供する。
