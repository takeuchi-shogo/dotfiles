# 用語 (Language)

このスキルが使う共通語彙。「コンポーネント」「サービス」「API」「境界」に流れない。

## 用語定義

**モジュール (Module)**: インターフェースと実装を持つもの。粒度を問わない (関数 / クラス / パッケージ / 層をまたぐスライス)。
_避ける_: ユニット、コンポーネント、サービス。

**インターフェース (Interface)**: 呼び出し側が正しく使うために知る必要のある全て。型シグネチャ + 不変条件 + 順序制約 + エラーモード + 必須設定 + パフォーマンス特性。
_避ける_: API、シグネチャ (型レベルだけを指して狭い)。

**実装 (Implementation)**: モジュールの中身。アダプターと区別する。小さなアダプター + 大きな実装 (Postgres リポジトリ) と、大きなアダプター + 小さな実装 (in-memory fake) は別物。

**深さ (Depth)**: インターフェースでのレバレッジ。学ぶインターフェース量に対して呼び出し側 (またはテスト) が得られる挙動の量。

- **深い**: 小さなインターフェースの裏に大量の挙動。
- **浅い**: インターフェースが実装と同じくらい複雑。

**シーム (Seam, Michael Feathers の用語)**: そこを編集せずに挙動を変えられる場所。モジュールのインターフェースが住む場所。シームをどこに置くかは、その奥に何を置くかとは別の設計判断。
_避ける_: 境界 (DDD の bounded context と衝突)。

**アダプター (Adapter)**: シームに座ってインターフェースを満たす具体物。役割 (どのスロットを埋めるか) を表し、中身を表すのではない。

**レバレッジ (Leverage)**: 深さから呼び出し側が得るもの。学ぶインターフェース量に対する能力。1 つの実装が N 呼び出し + M テストで支払い直す。

**局所性 (Locality)**: 深さから保守側が得るもの。変更・バグ・知識・検証が呼び出し側に散らばらず 1 箇所に集まる。一度直せば全部直る。

## 原則

- **深さはインターフェースの性質、実装の性質ではない**: 深いモジュールは内部で小さくモック可能・差し替え可能な部品で構成されていてよい。それらはインターフェースの一部ではない。モジュールは **内部シーム** (実装内部のテスト用) と **外部シーム** (インターフェース上) の両方を持てる。
- **削除テスト (Deletion test)**: モジュールを削除すると想像する。複雑性が消えればモジュールは何も隠していなかった (pass-through)。複雑性が N 呼び出し側に再出現するなら、モジュールは稼いでいた。
- **インターフェース = テスト面**: 呼び出し側もテストも同じシームを越える。インターフェースの **奥** をテストしたいなら、モジュールの形がたぶん間違っている。
- **アダプター 1 個 = 仮説のシーム、2 個 = 本物のシーム**: 何かが横断して変わらないならシームを導入するな。

## 関係

- **モジュール** は **インターフェース** を 1 つ持つ
- **深さ** は **モジュール** の性質、**インターフェース** に対して測る
- **シーム** は **モジュール** の **インターフェース** が住む場所
- **アダプター** は **シーム** に座って **インターフェース** を満たす
- **深さ** は呼び出し側に **レバレッジ**、保守側に **局所性** をもたらす

## 棄却した枠組み

- **深さ = 実装行数 / インターフェース行数 (Ousterhout の素朴解釈)**: 実装を膨らませることに報酬を与えてしまう。レバレッジとしての深さを使う。
- **「インターフェース」 = TypeScript の interface キーワード / クラスの public メソッドのみ**: 狭すぎる。ここでのインターフェースは呼び出し側が知るべき全事実を含む (不変条件・順序・エラー・性能特性等)。
- **「境界 (boundary)」**: DDD の bounded context と衝突する。**シーム** か **インターフェース** と呼ぶ。
