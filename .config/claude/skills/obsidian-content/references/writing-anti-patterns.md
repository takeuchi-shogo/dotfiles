# Writing Anti-Patterns — AI臭い文章を避けるためのガイド

obsidian-content の品質チェック（Step 4）で参照する。
生成した文章にこれらのパターンが含まれていたら修正する。

Source: https://every.to/guides/ai-style-guide

## 文構造のアンチパターン

| Bad | Good | なぜ Bad か |
|-----|------|------------|
| 「Xではなく、Yである」構文を多用 | Yを直接述べる | 否定から入る足場は冗長。読者はYだけ知りたい |
| 「実は」「たぶん」「ちょっと」の多用 | 削除。知的な留保が必要な場合のみ残す | ヘッジは自信のなさを演出し、文を弱める |
| 修辞疑問文で埋める（「〜ではないでしょうか？」） | 断言に変換するか削除 | フィラーとして使うと読者の注意が散る |
| 冒頭3段落以上かけてやっと本題 | 最初の文で摩擦・テンションを提示 | 読者は3行で離脱する |
| 要約して終わる結論（「以上のように〜」） | 視点を拡張するか、問いを投げて終わる | エネルギーが萎む。読後感がない |
| 汎用的な接続詞（「次に」「また」「さらに」） | 削除するか、具体的な繋がりを示す | 読者は論理的接続を文脈から読み取れる |
| 「研究によると」（出典なし） | 研究名・著者・年を明記。できないなら削除 | 権威の借用は信頼を損なう |

## 抽象的・ジェネリック表現

| Bad (AI臭い) | Good (具体的) | 原則 |
|--------------|--------------|------|
| 「ワークフローを最適化する」 | 「Claude が5つのPRを出す間にコーヒーを飲んだ」 | 動作する場面が見える表現にする |
| 「コスト効率が良い」 | 「月4万円で年4000万円分の仕事を代替」 | 数字で語る |
| 「スケーリングの課題に直面した」 | 「Gmail のレート制限で2,000件のメール操作がブロックされた」 | 固有名詞と数値で具体化 |
| 「感動的な体験でした」 | 「手が震えて、画面が滲んで見えた」 | 身体感覚で描写する |
| 「革新的なソリューション」 | 何が・どう・誰のために変わったかを書く | 形容詞ではなく事実で革新性を示す |
| 「シームレスな統合」 | 具体的に何と何がどう繋がるかを書く | バズワードは情報量ゼロ |

## 構成のアンチパターン

| Bad | Good | なぜ Bad か |
|-----|------|------------|
| 業界分析から始める（個人の賭けがない） | 自分の体験・摩擦から始める | 読者が感情移入できない |
| プロセス説明に感情の弧がない | 失敗→発見→変化の弧を織り込む | ドキュメントではなく物語にする |
| 理論→実践の順 | 実践（具体例）→理論（なぜ機能するか）の順 | 具体が先、抽象が後。帰納的に書く |
| 文脈/歴史から始める | 摩擦/問題から始める | 「まず背景ですが…」は読者が最も興味を失う開き方 |
| 整いすぎた対称的フレーズ | 少し崩す、生の言葉を混ぜる | 過度に磨かれた文は人工的に感じる |
| 自分の文に酔った感じ | 読者の利益を優先する | 「うまいこと言った」感は読者を引かせる |

## チェックリスト（生成後セルフレビュー用）

生成したコンテンツに以下を適用する:

- [ ] 冒頭1-2文で読者の注意を引いているか（摩擦/テンション）
- [ ] 抽象的な形容詞（革新的、シームレス、最適化）が3つ以上ないか
- [ ] 「Xではなく、Y」構文が2回以上繰り返されていないか
- [ ] 結論が単なる要約になっていないか
- [ ] 具体的な数字・固有名詞・場面描写があるか
- [ ] ヘッジ（実は、たぶん、ちょっと）を不必要に使っていないか
- [ ] 出典なしの「研究によると」がないか
- [ ] 読者にとっての価値（感情 or 行動変容）が明確か
