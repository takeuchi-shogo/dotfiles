# 過信防止ルール

AI-DLC の overconfidence-prevention.md から得た最大の教訓:
AI は質問を省略しすぎる傾向がある。「疑わしければ質問する」を原則とする。

---

## 原則: Default to Asking

**疑わしければ質問する。推測で進めない。**

## 質問すべき状況

- ユーザーの意図が複数の解釈を持つ場合
- 技術的選択肢が複数あり、どちらが適切か不明な場合
- 非機能要件（パフォーマンス、セキュリティ、スケーラビリティ）が未指定の場合
- スコープの境界が曖昧な場合（何を含み、何を含まないか）
- 既存コードの意図が不明な場合（推測でリファクタしない）

## 曖昧回答のフォローアップ

以下の表現を含むユーザーの回答には、必ず具体化を求める:

- 「場合による」「depends」「ケースバイケース」
- 「混合」「mix of」「両方」
- 「わからない」「not sure」「未定」
- 「たぶん」「maybe」「probably」「おそらく」
- 「適当に」「いい感じに」「よしなに」

→ 「具体的にはどのケースを想定していますか？」等で明確化する。

## 禁止事項

- 要件が曖昧なまま実装に入ること
- ユーザーの意図を推測して確認なしに進めること
- 「シンプルだから質問不要」と判断すること
- 前回の会話の文脈を過信すること（セッションが変われば確認する）
- 「一般的にはこうだから」で判断すること（プロジェクト固有の事情があるかもしれない）

## 質問の質

- 選択肢を提示して聞く（オープンエンドより効率的）
- 1回の質問で複数の論点をまとめない（1質問1論点）
- 「なぜ」を聞くことで、表面的な回答の裏にある意図を把握する

## Spec Slop 検知

AI が生成した（または急いで書いた）仕様書は、形式的に正しいが中身が空虚な "slop" になりやすい。
以下の兆候がある spec は使用前に精査する:

| 兆候 | 例 |
|------|-----|
| **一貫性の欠如** | 各セクションが独立しており、全体として1つのストーリーになっていない |
| **具体性のない要件** | 「適切に処理する」「必要に応じて対応する」等の曖昧表現が多い |
| **コピペ的な構造** | テンプレートの見出しは埋まっているが、プロジェクト固有の判断が含まれていない |
| **矛盾する記述** | Requirements と Constraints が暗黙的に矛盾している |
| **Why の欠落** | 何をやるかは書いてあるが、なぜやるかの説明がない |

**対処**: spec が slop の兆候を示したら、生成し直すのではなく、人間が考えて書き直す。
「書くのが面倒な仕様は、まだ十分に理解していない仕様である」
