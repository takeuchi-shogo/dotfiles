# 存在 ≠ 十分 原則

> Schwartz "Vibe Physics" (2026-03) の吸収時に発見した構造的盲点:
> 「仕組みが存在する」と判定した時点で深掘りを止め、
> 具体的な弱点・盲点・パラメータ改善の余地を見逃した。

---

## 原則

**分析・監査・ヘルスチェックにおいて、「存在する」は「十分である」の証拠にならない。**

存在を確認した後、以下の3問で十分性を評価する:

1. **カバレッジ**: 想定されるケースのうち、どれだけがカバーされているか？
2. **深さ**: 表面的な記述か、具体的な判断基準・手順・例を含むか？
3. **鮮度**: 現在のコードベース・ワークフロー・知見を反映しているか？

## 適用場面

| 場面 | 存在チェック（不十分） | 十分性チェック（必要） |
|------|----------------------|----------------------|
| `/absorb` のギャップ分析 | 「この仕組みがある → Already」 | 「記事の失敗事例をこの仕組みで検出できるか？」 |
| `/init-project` の既存構造分析 | 「CLAUDE.md がある → OK」 | 「WHY/WHAT/HOW の3層が具体的に書かれているか？」 |
| `/check-health` のドキュメント鮮度 | 「ファイルが存在し更新日が新しい → OK」 | 「内容が現在のコード・設定と一致しているか？」 |
| `/ai-workflow-audit` | 「skill がある → カバー済み」 | 「description が正確で、内容がベストプラクティスに沿っているか？」 |
| `/improve` の改善提案 | 「設定が存在する → スキップ」 | 「最新の learnings データと照合して改善余地はないか？」 |

## 実践ガイド

### 2パス分析パターン

分析・監査系の処理では、以下の2パスを標準とする:

**Pass 1: 存在チェック** — 仕組み・ファイル・設定が存在するかを確認し、Gap/Already/Partial/N/A を判定する。

**Pass 2: 十分性チェック** — Already と判定した項目について、カバレッジ・深さ・鮮度を評価し、強化可能な項目を特定する。

### 「十分」の判定基準

| レベル | 基準 |
|--------|------|
| **十分** | 想定ケースの 80% 以上をカバーし、具体的な判断基準を含み、直近30日以内の知見を反映 |
| **改善可能** | 存在するが、カバレッジ・深さ・鮮度のいずれかに明確な改善余地がある |
| **形だけ** | ファイルは存在するが、中身がテンプレートのまま or 抽象的すぎて実用性が低い |
