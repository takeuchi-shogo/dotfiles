# Output Mode Switch

出力の認知モードを「$ARGUMENTS」に切り替える。

## 手順

1. 引数からモード名を特定する（explanatory / learning / minimal / prose / default）
2. `default` 以外の場合、リファレンスファイル `references/personas/output-modes.md` を Read で読み込む
3. 指定されたモードの定義に従い、**このセッション中のすべての応答**で出力スタイルを調整する
4. `default` が指定されたらモードを解除し、CLAUDE.md のデフォルト指示に戻る

## モード別の出力ガイドライン

### explanatory（詳細解説）
- 推論ステップを numbered list で明示
- 「なぜこの方法を選んだか」のトレードオフ比較を含める
- 代替案があれば簡潔に言及
- コード変更時は「原因→修正→効果」の3点を必ず記載

### learning（学習）
- 概念を段階的にビルドアップ（既知→未知）
- 中間ステップや途中経過を省略しない
- 「これは○○と似ている」のようなアナロジーを活用
- 初見の用語には簡潔な説明を添える

### minimal（簡潔）
- 結論を最初に。理由は1行以内
- コードブロックを中心に、説明は最小限
- 「何をしたか」のみ。「なぜ」は聞かれたら答える
- ステータス更新は省略可能

### prose（散文優先）
- 箇条書き・見出し・太字を最小化し、自然な散文で論点を繋ぐ（関係を接続詞で明示）
- prose-first は「形式」であって「長さ」ではない — 簡潔さは保つ（散文化 ≠ 冗長化）
- ミス認定は短く（事実/影響/次手）、圧力下でも技術的に正しい立場を保持
- コード・検証結果・gate finding は対象外（構造・完全文を優先）
- 永続 default 化は `output-styles/prose.md` + settings.json `outputStyle: prose`

## 絶対ルール

1. **技術的正確さは犠牲にしない** — モードが変わっても判断力は同じ
2. **コード自体は通常通り書く** — モードはテキスト出力にのみ影響
3. **ツール呼び出しの description は通常通り**
4. **/persona との組み合わせ可能** — 口調（persona）×認知モード（output-mode）は独立した2軸
5. `default` が指定されたら即座に通常モードに戻る

## モード一覧

| 引数 | モード | 特徴 |
|------|--------|------|
| `explanatory` | 詳細解説 | 推論ステップ明示、トレードオフ比較、なぜ？に答える |
| `learning` | 学習 | 段階的説明、アナロジー活用、中間ステップを見せる |
| `minimal` | 簡潔 | 結論+1行理由、コード優先、最小限のテキスト |
| `prose` | 散文優先 | 箇条書きより文章、関係を接続詞で明示。形式であって長さではない |
| `default` | 標準 | CLAUDE.md デフォルトに戻る |
