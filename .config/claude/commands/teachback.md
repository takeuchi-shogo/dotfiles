# /teachback — このセッションの作業を教え返す

このセッションで **Claude (あなた) が行った作業** を、ユーザーが深く理解できるよう教師役として検証する。「エージェントの作業をループに入って完全に理解する」ための opt-in クロージングモード。

引数: `$ARGUMENTS`
- (なし) — 通常モード (ソフトゲート)。チェックリスト主要項目が未確認のまま終了しようとしたら、残り項目を提示して一度確認する。最終的な終了判断はユーザーに委ねる (受動的には終わらせない)
- `--strict` — hard gate。理解が検証されるまでセッションを終えない (摩擦高、明示要求時のみ)

> 対象は **外部記事** ではなく **このセッションの変更** (問題・解決・設計判断・エッジケース)。外部記事の理解は `/deep-read`、CC セットアップ知識の自己テストは `/quiz` を使う。

## 進め方

賢く効果的な教師として振る舞う。一度に全部やらず、**段階的に**進める。各段階で習得を確認してから次へ進む。high level (動機) と low level (ビジネスロジック・エッジケース) の両方を扱う。

### 1. running チェックリストを作る

ユーザーが理解すべき項目を md チェックリストで管理し、進行に応じて更新する。最低限カバーする 3 階層:
1. **問題** — 何が問題だったか / なぜ存在したか / 検討した分岐
2. **解決** — なぜその方法で解決したか / 設計判断 / エッジケース
3. **広い文脈** — なぜこれが重要か / 何に影響するか

### 2. まず再陳述させる (restate-first)

説明から入らず、**先にユーザーに現在の理解を述べてもらう**。そこからギャップを埋める。ユーザーは質問や `eli5` / `eli14` / `eli-intern` (インターン向け) の説明レベルを要求できる。

### 3. why を掘る

what と how だけでなく **why を繰り返し掘る** (なぜそうした → さらになぜ)。問題をよく理解することが最重要。

### 4. クイズで検証する

`AskUserQuestion` で記述式または選択式の問題を出す:
- **正答の位置は毎回シャッフルする**
- **全問提出までは答えを明かさない**
- 必要ならコードを提示する / debugger を使わせる

### 5. ギャップを埋める

回答が不正確なら、該当箇所を具体的に示して正しい理解へ導く。チェックリストを更新する。

## 完了条件

- 通常モード (ソフトゲート): チェックリストの主要項目をユーザーが説明できたら締める。未確認のまま終了の兆候があれば、残り項目を提示して「このまま終えるか / 続けるか」を一度確認する (受動的に終わらせない)
- `--strict`: チェックリスト全項目の習得をユーザーが実証するまでセッションを終えない

## Anti-Patterns

- 一度に大量の項目を説明する (段階的に、習得確認しながら進める)
- 再陳述させずに講義から入る (まずユーザーに述べさせる)
- why を掘らず what だけで終わる
- 答えを先に明かしてからクイズする
