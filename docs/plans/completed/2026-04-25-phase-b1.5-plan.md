## Background

Phase B1 Step 7 で bootstrap 5 ツール (git, sheldon, starship, mise, direnv) を nix 管理に移したが、以下で失敗し 9887dbe で rollback:

- direnv `checkPhase` が Determinate Nix + sudo + Apple Silicon 環境でハング
- `overrideAttrs (doCheck = false)` は Hydra cache 無効化 → 毎 switch で go-modules 再ビルド (5+ 分)

## Strategy Decision

Codex deep reasoning (gpt-5.4 xhigh) で 4 案を評価した結果:

| Strategy | Verdict |
|---|---|
| 1. Upstream patch (nixpkgs に `doCheck = lib.mkDefault false` PR) | 長期解だが blocker 解消には遅い |
| 2. Per-package overlay (`overlays/direnv.nix` で patches 側 test skip) | overrideAttrs よりマシだが patch 追従が SLO 違反 |
| 3. `nix profile` only (home-manager から切り出し) | `darwin-rebuild` rollback と canonical source を破壊 |
| **4. homebrew-in-nix (`nix-darwin.homebrew.brews`)** | **推奨**: 宣言性を保ったまま brew に委譲 |

**Selected**: Hybrid **4 + 1**
- 主戦略: Strategy 4 で 5 ツールを `nix-darwin.homebrew.brews` に宣言
- 補助戦略: Strategy 1 で direnv hang を upstream issue 化（merge 待ちはしない）

## Rationale

- 既存 master plan D9 で「Homebrew を nix-darwin 経由で宣言的に残す」方針が明記済み
- Strategy 4 は「妥協」ではなく「計画内の例外処理」
- canonical source が flake 側に一元化され、現状の「Brewfile + nix の二重管理」を解消

## Rollback Criteria

以下のいずれかで即 abort:

1. 1 ツールあたり調査/修正が **2 時間超**
2. warm `nix:switch` が **5 分超を 2 回連続**
3. hidden source rebuild、PATH 回帰、shell 起動劣化の再発

いずれかを踏んだら、そのツールは即 `brew-retain` に戻し、bootstrap set 全体の再 nix 化は次の upstream 改善まで凍結。
