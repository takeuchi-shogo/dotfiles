# 第23章 Agent-to-Agent Communication (A2A)（エージェント間通信）

> **一文サマリ:** A2A は、異種・異ベンダーのエージェント同士が発見・委譲・結果集約するための標準プロトコル(Google, 2025年4月 → Linux Foundation 寄贈)。MCP がエージェント↔ツールの**縦**なら、A2A はエージェント↔エージェントの**横**。
> **PDF参照:** §23, p.417-438

> **先に正直な scope 注記:** A2A は「異なるチーム/ベンダー/組織が作ったエージェントを相互運用する」ための enterprise 標準。**単一の個人リサーチエージェントには過剰**になりやすい。あなたの例題で本当に効くのは「収集役/選別役/要約役を1システム内で分業」する **Ch24 Multi-Agent**。本章は「将来、外部の専門エージェントに下請けを出す」「複数システムを繋ぐ」段になったときの参照として読む。原則(発見・委譲・信頼・監査)は Ch24 にも効く。

## 23.1 なぜエージェントは通信するか

単一の generalist は「知識の幅 vs 能力の深さ」の根本的緊張に直面する。A2A は**専門家のネットワーク**で解く。5つの力: ①**Cognitive Load**(各エージェントが扱える context は有限 → タスクを分解し各自は管理可能な context で動く)②**Specialization**(CodeAgent/LegalAgent/DataAgent を専門ツールで武装)③**Parallelism**(独立subtaskを複数エージェントに同時 fan-out)④**Fault Isolation**(1エージェント失敗で別へretry・縮退・人間へescalate、全体を壊さない)⑤**Delegation/Handoff**(PlannerAgent→ExecutorAgent→ReviewerAgent に必要な context だけ渡す)。

**A2A の5要件:** Discoverability(他エージェントとその能力を発見)/ Interoperability(別チーム/ベンダー製でも共通プロトコル)/ Asynchrony(長時間タスクが呼び出し側をブロックしない)/ Security(相互認証・認可境界)/ Observability(全メッセージ交換が追跡可能)。

## 23.2 Google A2A プロトコル

### Agent Card + Task ライフサイクル（発見の基盤）
**これは何か** — **Agent Card** は `/.well-known/agent.json` に置く機械可読 JSON manifest。何ができるか・どう認証するか・どこにタスクを送るかを広告(OpenAPI spec のエージェント版)。`name`/`description`/`url`/`capabilities`(streaming/pushNotifications)/`authentication`/`skills`(id/name/description/tags/examples/inputModes/outputModes) を含む。全作業は **Task** としてモデル化され状態機械を進む: `submitted`→`working`→(`input-required`)→`completed`/`failed`/`rejected`/`canceled`。
**いつ使う / なぜ要る** — 動的に専門エージェントを見つけて委譲したいとき。Agent Card で **capability-based routing**(URLハードコードでなく能力で選ぶ)が可能になる。
**最小実装**
```python
agent_card = {
    "name": "ResearchAgent",
    "description": "学術文献を検索し調査結果を統合する",
    "url": "https://research-agent.example.com/a2a",
    "capabilities": {"streaming": True, "pushNotifications": False},
    "authentication": {"schemes": ["Bearer"]},
    "skills": [{"id": "literature-search", "name": "Literature Search",
                "description": "トピックの学術論文を検索・要約",
                "tags": ["research", "literature"], "inputModes": ["text"],
                "outputModes": ["text", "data"]}],
}
# /.well-known/agent.json で公開 → オーケストレータが registry から取得して照合
```
**リサーチエージェントでの使いどころ** — 自分のリサーチエージェントを Agent Card 化すれば、他のオーケストレータから「文献調査の専門家」として呼ばれる。逆に外部の専門エージェント(統計解析・可視化)に下請けを出せる。
**落とし穴** — ①long-running task で SSE 接続を何時間も保持しない → **push notification**(webhook URL登録、Serverが状態をPOST)②`input-required` 状態(明確化が要る)を扱わないと task が宙づり。
**PDF参照** — §23.2, p.418-421

**設計哲学(5原則):** ①**Opaque execution**(相手の内部を覗かない、宣言された interface だけで対話 → GPT/Gemini/ルールベースを問わず真に異種なエコシステム)②**Enterprise readiness**(OAuth 2.0/JWT・監査ログ・規制対応がプロトコルレベルで最初から)③**Modality agnostic**(1メッセージにtext+binary+構造化JSONを混在)④**Simplicity via existing standards**(新transportを発明せず HTTP/JSON-RPC 2.0/SSE/gRPC を再利用)⑤**Async-first**(長時間操作が常態、push通知とpollingが第一級)。

**Streaming(SSE):** 逐次出力(長いレポート生成)は Server-Sent Events で `TaskStatusUpdateEvent`/`TaskArtifactUpdateEvent` をストリーム。**Auth:** Bearer(JWT/OAuth 2.0、scopeで権限制限)/ API key / mTLS / OpenID Connect。

> **Authorization Scope の強制(最重要):** タスクを受けたエージェントは「誰が呼んでいるか(認証)」だけでなく「何を要求してよいか(認可)」を検証する。ReportingAgent は読み取りクエリは任意の認証済みエージェントから受けても、書き込みは特定 OAuth scope 保持者だけに制限する。怠ると multi-agent で**権限昇格脆弱性**。

## 23.3 通信パターン

| パターン | レイテンシ | 向き |
|---|---|---|
| Request-Response | 低 | 短い・定義明確なsubtask |
| Streaming | 低(first token) | 長文生成・リアルタイムUI |
| Multi-Turn | 中 | 明確化が要る曖昧タスク(`input-required`) |
| Broadcast | 低 | 共有context配布 |
| Pub-Sub | 可変 | イベント駆動の反応的ワークフロー |
| Negotiation | 中〜高 | 資源制約下の計画(提案↔逆提案) |
| Auction | 中 | 動的負荷分散(入札で勝者にawardする) |

## 23.4 エージェント発見とルーティング

### Capability-based routing（URLでなく能力で選ぶ）
**これは何か** — **Agent registry**(Agent Card を索引する directory、centralized=単一権威だが SPOF / federated=ドメイン毎+横断検索でレジリエント)から、必要skillに合うエージェントを照会して選ぶ。意味的には「タスク記述をembedding → 各エージェントのdescriptionとcos類似度 → 最高スコアを選択」。
**いつ使う / なぜ要る** — エージェントURLをハードコードしたくないとき。新しい専門家が registry に増えれば自動で使える。
**最小実装**
```python
async def route(self, task_description: str) -> AgentCard:
    task_emb = await embed(task_description)
    agents = await self.find_agents(required_skill="*")        # registryから全取得
    scored = [(cosine(task_emb, await embed(a.description)), a) for a in agents]
    return max(scored, key=lambda x: x[0])[1]                  # 最も近いエージェント
```
**リサーチエージェントでの使いどころ** — 「統計解析が要る」subtaskを registry の DataAgent に動的ルーティング。負荷分散は round-robin / least-loaded / latency-aware / affinity-based(関連タスクを同一エージェントへ=cache活用)。
**落とし穴** — ①**version skew**(本番では各エージェントが別時期に更新される。v2.1向けに組んだオーケストレータが v1.3 のエージェントに当たる → 後方互換なメッセージ処理 + cross-version テスト、semantic versioning)②embedding routing は description の質に依存。
**PDF参照** — §23.4, p.423-424

## 23.5 メッセージ形式と context 受け渡し

メッセージ = `role`(user/agent) + 型付き `parts` のリスト。**Part 3種:** TextPart(`text`、自然言語)/ FilePart(`mimeType`+`uri`or`bytes`、文書・画像・音声・コード)/ DataPart(`data`、構造化JSON: ツール結果・スキーマ)。1メッセージにtext+file+dataを混在できる(マルチモーダル)。

> **Context Scoping(何を渡し何を秘すか、重要):** ①**Minimal Context**(sub-agentがタスク完了に必要な分だけ渡す → token・レイテンシ・情報漏洩リスクを削減)②**Summarized Context**(生の履歴でなく構造化サマリ: 目標・制約・決定・関連事実)③**Private State**(内部推論・中間ドラフト・ユーザPIIは原則 sub-agent に転送しない)④**Correlation ID**(`correlationId`/`workflow_id`/`span_id` を必ず渡し、sub-agent の行動を元ワークフローに辿れるようにする)。プロジェクトの「境界では最小限・内部はTrust」と同型。

## 23.6 協調プロトコル

点対点通信を超えた集団的意思決定:

| プロトコル | 仕組み | 向き |
|---|---|---|
| **Contract Net** | manager がタスクを broadcast → contractor が入札(時間/信頼度/コスト) → 最良に award | 動的タスク割当 |
| **Blackboard** | 共有黒板に部分解を投稿、他エージェントが価値を足せるとき貢献(opportunistic) | 解の道筋が事前に不明(科学的仮説生成・複雑デバッグ) |
| **Consensus** | majority voting / weighted(信頼度加重) / quorum(n中k一致、n-k失敗まで許容) / Delphi(匿名結果を見て再投票、anchoring減) | 複数エージェントの合意 |
| **Leader Election** | Bully/Ring を能力スコア交換に適応、最有能なエージェントをリーダーに | オーケストレータ障害時の再選出 |

## 23.7 A2A vs MCP（補完的、競合でない）

> **核の区別:** **MCP は縦**(エージェントを下方=DB/API/FS/コード実行へ伸ばす。推論するのはエージェントだけ、MCP端点は決定的サービス)。**A2A は横**(推論するエージェント同士を繋ぐ。両側が推論・計画・ツール利用できる知的アクター)。

| 観点 | MCP | A2A |
|---|---|---|
| 参加者 | Agent ↔ Tool/Resource | Agent ↔ Agent |
| 知性 | 片側(agent)のみ | 両側が知的 |
| state | 大抵 stateless tool call | lifecycle付き stateful task |
| 発見 | Tool manifest | Agent Card |
| レイテンシ | ミリ秒 | 秒〜分 |
| 用途 | 「Web検索」「SQL実行」 | 「専門家に委譲」 |

**使い分け:** リモート端点が**決定的関数**(DBクエリ・API・サンドボックス)→ MCP。リモート端点が**推論を要する**(曖昧な指示を解釈・判断・自前ツール・多ターン対話)→ A2A。**両方**: オーケストレータが A2A で専門エージェントに委譲し、各専門エージェントが MCP で自分のツールに触る。

```
        Orchestrator
       ╱ (A2A) │ (A2A) ╲ (A2A)         ← 横: エージェント間委譲
  CodeAgent  DataAgent  WritingAgent
   │(MCP)     │(MCP)      │(MCP)        ← 縦: 各自のツール接続
  Compiler  Database   Web Search
```
ツールは決して他エージェントに直接露出せず、**所有エージェントのI/F越し**だけ(transitive tool access なし=信頼境界の分離)。

## 23.8 セキュリティと信頼

**Identity:** JWT(署名付き、ID/issuer/expiry)/ mTLS(証明書、認証+暗号化)/ DID(分散ID、クロス組織で単一権威不在のとき)。**Integrity:** 全通信を TLS 1.3 / 機密ペイロードは E2E暗号(JWE、中間のLB/proxyが読めない)/ message signing(JWS、否認防止)。**Authz scopes:** OAuth 2.0 scope(`data:read`/`data:write`/`admin:config`)で「誰が何を要求してよいか」を境界化、受信側が `verify_authorization` で検証。**Audit:** 全 A2A 交換を構造化ログ(timestamp/workflow_id/span_id/parent_span_id/caller・callee identity/skill/scopes/outcome)。

> **Accountability Gap:** 委譲連鎖(A→B→C)で「誰が責任者か」が曖昧になる。AがBにファイル削除を頼みBが実行したら責任は? → 全交換に caller identity・task・使った認可token・timestamp・結果を記録。これがインシデント対応・コンプライアンス・デバッグに必須。dotfiles の "mutation は決定的コマンドで検証 / 監査ログ" と同根。

## 23.9 実装例: マルチエージェントのリサーチワークフロー（あなたの題材）

A2A の完全例は **ResearchOrchestrator**: 調査命題を分解 → 各 sub-question を ResearchAgent(専門家)に委譲 → 結果を統合。これはあなたが作る物の multi-agent 版。

```python
class ResearchOrchestrator:
    async def run(self, research_question: str) -> str:
        sub_questions = self._decompose(research_question)        # 1. 命題を小問に分解
        tasks = await asyncio.gather(*[                           # 2. 各小問を並列委譲
            self.research_client.send_task(
                message=Message(role="user", parts=[Part(type="text", text=q)]),
                metadata={"workflowId": self.workflow_id, "subQuestion": i})
            for i, q in enumerate(sub_questions)])
        done = await asyncio.gather(*[                            # 3. 全完了を待つ
            self.research_client.wait_for_completion(t) for t in tasks])
        failed = [t for t in done if t.status.state == "failed"]  # 4. 失敗を確認(握り潰さない)
        findings = [t.artifacts[0].parts[0].text for t in done    # 5. 結果を統合
                    if t.status.state == "completed"]
        return self._synthesize(research_question, findings)
# ResearchAgent 側は FastAPI で /.well-known/agent.json と /tasks/send を公開する A2A Server
```
`_decompose`(命題→小問)と `_synthesize`(小問結果→レポート)は本番では LLM。`workflowId` で全 sub-task を追跡。

> ただし**個人用なら**、この A2A(HTTP server/Agent Card/認証)のオーバーヘッドは過剰。同じ「分解→並列委譲→統合」を**1プロセス内の sub-agent**(Ch24、dotfilesなら Agent tool / Workflow)でやる方が軽い。A2A が要るのは「ResearchAgent を他人にも使わせる」「外部の有償専門エージェントに繋ぐ」段になってから。

## この章のまとめ

- A2A = 異種・異ベンダーのエージェント間の発見・委譲・集約の標準(横の協調)。**個人リサーチagentには過剰になりやすい — まず Ch24 の1システム内 multi-agent で**。
- **Agent Card**(`/.well-known/agent.json`)が発見の基盤、**Task** が状態機械(submitted→working→completed等)。capability-based routing で能力からエージェントを選ぶ。
- **A2A(横、両側知的)と MCP(縦、片側知的+決定的ツール)は補完的**。本番は両用: A2Aで委譲、各自MCPでツール接続。
- **Context scoping**(最小・要約・private・correlation ID)と **security**(identity/scope/audit)が必須。Accountability gap を監査ログで埋める。
- 協調プロトコル(Contract Net/Blackboard/Consensus)が集団意思決定を構造化。
- **Open questions:** 階層内の指示衝突 / 学習されたルーティング / agent間 prompt injection / context のプライバシー境界 / 大規模での一貫した global state。

**次章 → Ch24 Multi-Agent Systems（協調層）。** A2A の「通信」を超え、複数エージェントを**1つのシステムとして設計・協調・デバッグ**する層へ。あなたのリサーチエージェントの分業はここが本命。
