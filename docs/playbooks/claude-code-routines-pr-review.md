# Claude Code Routines — PR Review & Auto-Fix

Claude Code Routines ([docs](https://code.claude.com/docs/ja/routines)) で PR レビューと安全な自動修正を回す playbook。
2 つの routine を組み合わせる: `pull_request.opened` で即時レビュー、毎朝 9 時に未レビュー PR を総ざらい。

## When To Use

- PR を開いた瞬間に最低限の自動レビューと lint/format/typo の機械的修正を入れたい
- 朝の最初のタスクとして、前日からの未レビュー PR をまとめてレビュー済みにしておきたい
- 自分が触っていないリポジトリでも、自動レビュー bot 的な役割を低コストで動かしたい

## Prerequisites

- Pro / Max / Team / Enterprise いずれかのプラン
- Claude Code on the web が有効
- `pull_request.opened` trigger を使う場合は Claude GitHub App をルーティン作成画面からインストール済み
  - `/web-setup` だけでは webhook が来ないので注意
- レビュー対象リポジトリを routine に登録済み
- ブランチ push 設定: **Allow unrestricted branch pushes は OFF のまま**で OK（autofix は `claude/` プレフィックスブランチに作る）

## Auto-Fix Safety Boundary

両 routine 共通で守る安全境界。

### 自動修正してよい（mechanical fix only）

- Linter / Formatter 違反: `eslint --fix`, `prettier --write`, `biome check --apply`, `gofmt`, `ruff --fix`, `cargo fmt`
- 未使用 import の削除（自分が見える範囲のみ、re-export は触らない）
- import 順序の修正
- typo（コメント・docstring・log message・テスト名・変数名で意味が一意なもの）
- 型注釈の明らかな漏れ（修正方法が一意な場合のみ）

### 自動修正しない（レビューコメントのみ残す）

- ロジック変更を伴うもの全て
- セキュリティ修正（誤った fix が新しい脆弱性を生む）
- パフォーマンス改善
- API シグネチャ・型定義の変更
- テスト追加（intent が不明）
- dead code 削除（実は使われているかも）
- 振る舞いが変わる可能性が 1% でもあるもの

### 修正フロー

1. `claude/auto-fix-pr-<N>-<YYYYMMDD-HHmmss>` ブランチを PR HEAD から作成
2. Formatter → Linter `--fix` → typo → 未使用 import の順で適用
3. 必ずテスト実行: `package.json` / `Makefile` / `Taskfile.yml` / `justfile` から test スクリプトを検出して実行
4. テスト失敗 → ブランチ破棄、要約に abort 理由記載
5. テスト成功 → conventional commit (`chore: auto-fix lint/format/typo`) + push
6. 元 PR に通知コメント（修正内容のサマリ + ブランチリンク）。自動マージはしない

## Routine 1: `pull_request.opened` トリガー

トリガーされた 1 PR を対象に、レビュー + 自動修正ブランチ作成までを一気に実施する。

### Prompt

```text
あなたはこの PR の自動レビュアー兼自動修正担当です。レビューと、機械的に安全な修正をブランチ作成して通知してください。自己完結で完走、対話確認なし。

# Phase 1: 把握
1. `gh pr view --json number,title,body,isDraft,author,headRefName,additions,deletions,files` で対象特定
   - `isDraft: true` なら `gh pr comment` で「Draft のためスキップ」と残して終了
2. `gh pr diff` で差分を読む。差分内ファイルのみが評価対象
3. CLAUDE.md / AGENTS.md / PULL_REQUEST_TEMPLATE.md があれば事前読み込み
4. `gh pr checkout <number>` でローカル取得

# Phase 2: レビュー
観点（優先度順）:
- Correctness: ロジック誤り、境界値、nil/空配列、race、エラー握り潰し
- Security: 入力検証、認可漏れ、SQLi/XSS/SSRF/path traversal、シークレット混入
- Test: 新規ロジックのテスト不在、実装と同義反復テスト
- Perf: N+1、O(n²)、不要 allocation（ホットパスのみ）
- Style: 既存パターンからの逸脱のみ

重要度ラベル: blocking / should / nit / question

# Phase 3: 自動修正
1. `claude/auto-fix-pr-<N>-<timestamp>` ブランチを PR HEAD から作成
2. 適用順:
   a. Formatter 実行（prettier / biome / gofmt / ruff format / cargo fmt のうち存在するもの）
   b. Linter --fix 実行（eslint --fix / biome check --apply / ruff --fix）
   c. typo 修正（コメント・docstring・log message・テスト名で意味が一意なもののみ）
   d. 未使用 import 削除
3. テスト実行（package.json scripts / Taskfile.yml / Makefile / justfile から検出）
4. テスト失敗時: ブランチを破棄、レビュー要約に「自動修正候補ありだがテスト失敗のため abort」と記載
5. テスト成功時:
   - conventional commit: `chore: auto-fix lint/format/typo` 形式
   - `git push origin claude/auto-fix-pr-...`
   - 元 PR に通知コメント（修正内容のサマリ + ブランチリンク）

# Phase 4: 投稿
1. blocking と should はインラインコメント (`gh api repos/:owner/:repo/pulls/:number/comments`)
2. 要約コメントを `gh pr comment` で 1 本投稿:

   ## 自動レビュー要約
   - 規模: +X / -Y 行, N ファイル
   - blocking: N / should: N / nit: N / question: N
   - 自動修正: applied (branch: `claude/auto-fix-...`) / skipped (理由) / none

   ### Blocking
   - [path:line] 一文要約
   ### Should / Nit / Question
   - ...
   ### 良かった点
   - ...
   ---
   _Generated by Claude Code Routine_

# 禁止
- 安全境界外の修正（ロジック・セキュリティ・パフォーマンス・テスト追加・削除）
- 差分外ファイルへの修正波及
- テスト未実行の commit
- 元 PR ブランチへの直接 push（`claude/` プレフィックスのみ使用）
- 「LGTM」だけのコメント
- 確信度の低い指摘
```

### Trigger Setup

- Trigger type: GitHub event → `pull_request.opened`
- Filter 推奨: `draft = false`（ドラフトを除外、プロンプトでも二重チェック）
- 自分が author の PR をルーティンがレビューする場合は filter で `author != <自分>` を入れるか、プロンプトで除外

## Routine 2: 毎日 9 時 schedule

open / 非 Draft / 未自動レビューの PR を最大 5 件処理し、Routine 1 と同じ安全境界で autofix を提案する。

### Prompt

```text
9 時の定時ジョブです。open / 非 Draft / 未自動レビューの PR を全件レビューし、機械的に安全な修正をブランチ作成して通知してください。自己完結で完走、対話確認なし。

# Phase 1: 対象抽出
1. `gh pr list --state open --draft=false --json number,title,author,reviewDecision,comments,updatedAt --limit 50`
2. 除外:
   - author が自分自身
   - 直近 24h に「Generated by Claude Code Routine」を含むコメントが存在
   - `reviewDecision: APPROVED`
3. `updatedAt` 昇順で最大 5 件処理、残りは「次回繰越し N 件」と最終出力

# Phase 2: 各 PR の処理ループ
PR ごとに以下を実行:

## 2-1. レビュー
- `gh pr checkout <number>` → `gh pr diff` 読解
- 観点・重要度ラベルは Routine 1 と同一（Correctness/Security/Test/Perf/Style、blocking/should/nit/question）

## 2-2. 自動修正
1. 既存 `claude/auto-fix-pr-<N>-*` ブランチが `git branch -r` で見つかったら autofix は skip（冪等性）
2. `claude/auto-fix-pr-<N>-<timestamp>` ブランチを PR HEAD から作成
3. Formatter → Linter --fix → typo → 未使用 import の順で適用
4. テスト実行（リポジトリのスクリプトを検出して実行）
5. テスト失敗 → ブランチ破棄、要約に abort 理由記載
6. テスト成功 → conventional commit + push + 通知コメント
7. 別 PR を作らず、元 PR にコメントで通知（ブランチリンクのみ）

## 2-3. 投稿
- blocking / should はインラインコメント
- 要約コメント:

   ## 日次自動レビュー (YYYY-MM-DD)
   - blocking: N / should: N / nit: N / question: N
   - 自動修正: applied (branch: `claude/...`) / skipped (理由) / none
   ### Blocking / Should / Nit / Question
   ### 良かった点
   ---
   _Generated by Claude Code Routine_

# Phase 3: 最終標準出力
処理済み: N 件
- #123 (blocking: 1, should: 2, autofix: applied)
- #124 (clean, autofix: none)
- #125 (autofix: aborted - test failed)
スキップ: M 件（理由別）
次回繰越し: K 件

# 禁止
- 安全境界外の修正
- 同じ PR に 24h 以内の二重レビュー（冪等性）
- 同じ PR への二重 autofix ブランチ作成
- GitHub API レート制限時は途中停止 + 標準出力に明記
- 確信度の低い指摘
```

### Trigger Setup

- Trigger type: Schedule → daily, 09:00 (local timezone)
- スタガーで 09:00〜09:15 にずれる想定
- 1 回限り実行ではなく recurring schedule

## Routine 1 と 2 の違い

| 項目 | Routine 1 (PR open) | Routine 2 (Daily 9 AM) |
|---|---|---|
| 対象 | トリガーされた 1 PR | open 未レビュー全件（最大 5） |
| 列挙 | 不要（context で渡る） | `gh pr list` で自前抽出 |
| 冪等性 | 不要（1 回のみ発火） | 必須（24h 二重レビュー禁止 + autofix ブランチ重複防止） |
| Author 除外 | 任意（filter or プロンプト） | 必須（自 PR を弾く） |
| Draft | 早期スキップ | `--draft=false` で除外 |
| 要約タイトル | `## 自動レビュー要約` | `## 日次自動レビュー (日付)` |

## Operational Notes

- 緑ステータス ≠ 成功: routine 一覧の緑はインフラエラー無しの意味。最初の数日はセッション URL を開いて実投稿内容を必ず確認する
- テストランナー未検出時: `autofix: skipped (no test runner detected)` と要約に明記、レビューは継続
- Monorepo: workspace ごとに分岐するロジックを足す必要があるため、初期は単一パッケージリポジトリだけに routine を絞る
- CI 連動の発展: 自動修正ブランチ push 後の CI 結果まで追って通知コメントを更新する発展形があるが、初期は不要
- マージ責任: autofix ブランチを誰がマージするかは人間判断。自動マージはしない
- 日次実行上限: ルーティンはアカウントごとの日次実行上限がある。9 時 routine が 1 件、open trigger は PR 数次第。`claude.ai/settings/usage` で確認
- コスト: 1 回限り実行は日次ルーティン上限にカウントされないが、recurring は上限内。サブスクリプション使用量は通常通り消費

## Permission Notes

- Routine が呼ぶ `gh pr review` / `gh pr comment` / `gh api` は GitHub App の権限で動く（PR コメントは branch push と別権限）
- `Allow unrestricted branch pushes` は OFF のままで運用可能（autofix は `claude/` プレフィックスブランチに作るため）
- ON にすると元 PR ブランチへ直接 push できるが衝突リスクあり、推奨しない

## Minimum Validation

- 最初の 1 週間は毎日セッション URL を開いて投稿内容を目視確認
- false positive が出たら該当観点をプロンプトから削るか「確信度の低い指摘は出さない」を強調
- autofix ブランチがマージされなかった場合は手動で削除、もしくは routine 側で 7 日経過したブランチを掃除する補助 routine を追加
