# AIエージェントのHuman-in-the-Loop評価を深化させる — 分析レポート

## Source Summary

**著者**: 澁井祐介（LayerX Ai Workforce事業部R&Dチーム）
**主張**: 個々のHITLメトリクスの「計測」を超え、2つの分析軸を導入することで実践的な意思決定ツールへ昇華させる:
1. **評価の非対称性** — 見逃しと過検出は等価ではなく、損失が異なる
2. **評価の総体性** — 個別最適ではなく、システム全体の文脈で評価すべき

**手法**:
- 非対称損失関数: L(Δ) = α·|Δ|^γ（下振れ）vs β·|Δ|（上振れ）、α > β, γ > 1
- カテゴリ別 Fβ スコア: 高リスク β≥3（Recall重視）、低リスク β=1
- タイミング非対称性: 遅延は指数ペナルティ c_late·|Δt|^δ
- HITL 発生パターン4分類: 集中型(Gini>0.6) / 分散型(CV<1) / フロントローディング型(FLI>0.7) / ランダムバースト型(B>0.3, CV>1)
- HITL 間の相互作用: 依存型カスケード / 競合型干渉（判断疲労） / 補完型シナジー

**根拠**: 見逃し1件の損害（本番DB破壊等）は不要なHITL 10件分の人件費を容易に超える。個別メトリクスが良好でも後半でコンテキスト疲労により未確認承認率が急上昇しうる。

**前提条件**: HITLを既に運用中。リスクカテゴリ分類が可能。過去のインシデントデータでパラメータ校正ができる。

## Plan

### Task 1: 非対称損失の設計原則を workflow-guide.md に追記 (S)

**対象**: `.config/claude/references/workflow-guide.md`
**内容**: ワークフローのゲート設計セクションに「非対称損失の原則」を追加
- 見逃し（HITL すべきだったのにスルー）は指数的損失
- 過検出（不要な HITL）は線形コスト
- カテゴリ別パラメータ目安: セキュリティ/DB変更 α=高、ドキュメント修正 α=低

### Task 2: レビューゲートにリスクカテゴリ別の評価方針を導入 (M)

**対象**: `.config/claude/references/workflow-guide.md`, review 関連エージェント定義
**内容**: 変更対象のリスクカテゴリに応じてレビューの厳格度を変える方針を定義
- High Risk（セキュリティ、DB migration、本番設定）: Recall 重視（β≥3 相当）— 見逃し許容度ゼロ
- Medium Risk（ビジネスロジック、API）: バランス（β=2 相当）
- Low Risk（ドキュメント、テスト、コメント）: Precision 重視（β=1 相当）
**依存**: Task 1

### Task 3: HITL 発生パターン分類を meta-analyzer に追加 (M)

**対象**: `.config/claude/agents/meta-analyzer.md`, AutoEvolve 関連
**内容**: セッションデータからHITL発生パターンを分類する分析観点を追加
- Gini 係数（集中度）、変動係数 CV、Burstiness 指標、FLI（フロントローディング指数）
- パターン別の改善示唆（集中型→自動化候補、バースト型→判断疲労リスク）
**依存**: なし

### Task 4: レビュー指摘の accept/reject フィードバックループ設計 (M)

**対象**: `docs/specs/` に spec を新規作成
**内容**: code-reviewer の指摘に対するユーザーの accept/reject を記録し、精度を追跡する仕組みの仕様策定
- NEEDS_FIX の指摘ごとに accept/reject/partial をログ
- カテゴリ別の false positive 率を算出
- AutoEvolve の Analyze フェーズで精度データを活用
**依存**: なし（spec 策定のみ、実装は別セッション）
