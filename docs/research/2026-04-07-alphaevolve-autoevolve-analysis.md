## Source Summary

**主張**: LLMと自動評価器を組み合わせた進化的エージェントにより、複雑な数学およびコンピュータサイエンス問題のアルゴリズム最適化が可能。Gemini Flash(探索) + Pro(最適化) の二層構造で、データセンター効率化から未解決数学問題まで幅広い実用成果を実証。

**手法**:
1. 二層LLMアンサンブル — Flash(幅広い探索) + Pro(深い最適化)
2. 進化的プログラミングループ — 生成→評価→選別→コンテキスト累積の反復(数百世代)
3. 自動評価メカニズム — スカラー値のユーザー定義評価関数、即時フィードバック
4. プロンプト組立エンジン — 成功アルゴリズムからコンテキストを動的構築
5. 候補データベース — 世代管理、高スコア解を次世代の親として再利用
6. 島モデル（多集団並行進化）— 複数独立集団を並行進化、定期的に移住で優良個体を交換
7. 差分パッチ生成 — 全体書き換えではなく親コードへの差分パッチをLLMに生成させる

**根拠**: Borg スケジューリング最適化で全世界計算資源の0.7%を回収、FlashAttention 32.5%高速化、56年ぶりの Strassen 法改善（4×4複素行列乗算を48スカラー積）、50以上の未解決数学問題で75%が既知最良解と同等、20%で改善。

**前提条件**: アルゴリズムとして記述可能で、スカラー値の自動評価関数が定義できる問題が対象。決定的評価 × 高イテレーション × 数学/CS ドメインで最も有効。

## Refine Phase (Codex + Gemini)

### Codex 批評の主要指摘
- **見落とし**: 島モデル、差分パッチ、親選択の多様性、マルチオブジェクティブ評価
- **過小評価**: モデルルーティングと候補DBを「強化不要」→「強化可能」に格上げすべき
- **前提の誤り**: イテレーション数 100x スケール差、評価関数の性質の違い (決定的 vs 確率的)、ドメインの違い (アルゴリズム vs ハーネス設定)

### Gemini 補完の主要知見
- 外部採用は限定的、計算コスト高
- 制約: ドメイン特異性、評価メトリクス脆弱性、世代間多様性維持
- 代替: HITL進化、Pareto進化、Verifiable Code Evolution

### ドメイン適合性の結論
AlphaEvolve の手法の直接移植ではなく、設計パターンの翻案が正しいアプローチ。
